# uitk — Changelog

## 2026

- **2026-06-19 DRY** — **`BridgeSlotsBase` now ships a default `header_init`, so bridge / workflow slots declare their header menu as *data, not code*.** Every DCC-bridge and workflow slot used to hand-roll the same `header_init` (a "Utilities" separator + `menu.add(...)` buttons + `set_help_text(fmt(...))`). That boilerplate is now a single default on `BridgeSlotsBase`, driven by two class-attr hooks: `HEADER_MENU_ITEMS` — a tuple of `(label, objectName, tooltip, handler_method_name)`, defaulting to Open Templates / Refresh / Clear Log — and `HELP_SPEC` — the `fmt()` kwargs dict (or `None`). A bridge with a different menu sets the tuple (unity → Open Unity Project; rizom → UV-editor + scripts); a panel that needs runtime-computed items overrides the `header_menu_items()` hook (the photogrammetry panels, whose tooltips carry the engine label). Help is the `help_spec()` hook (default returns `HELP_SPEC`). ~9 subclasses across mayatk / blendertk / extapps dropped their `header_init` for a data declaration. New `test_bridge_slots_header.py` (6) pins the mechanism: default items + wiring, custom override, dynamic method override, no-help-when-absent, and a missing-handler error.
- **2026-06-19 Feature/Fix** — **Preset combos now remember the active preset across sessions, flag unsaved edits, and Refresh actually reverts (`PresetManager`, `wire_combo`, `ComboBox.current_text_suffix`, `bridge/slots.py`).** Three related preset gaps, all rooted in the combo having *no notion of an "active preset"* — it drove everything off the transient `currentIndex` and persisted nothing:
  - **Active preset restored between sessions.** `PresetManager.active_preset` (backed by `pythontk.PresetStore.active`'s `.active` sidecar) records the selected preset. On `wire_combo`, the selection is restored **with signals blocked — no values are re-applied** (the widgets restore themselves from QSettings; only the *selection* needs restoring, which is the whole point). The preset combo was never registered with `StateManager`, so it previously persisted nothing; and persisting the **name** (not the combo *index*) is immune to the list reordering that rename/add/delete cause.
  - **Modified ("dirty") marker.** `is_modified()` compares the live values (`_capture_values`, the shared SSoT `save` also uses) against the active preset's stored snapshot over the preset's keys (overlay semantics). When they diverge the combo paints a `" *"` suffix on the *displayed* current text via the new reusable `ComboBox.current_text_suffix` (item data / `itemText` stay the raw name, so load/rename/delete are unaffected). `refresh_modified_state()` + `on_modified_changed()` observers fire only on a transition; `connect_value_widgets()` auto-wires the marker for menu/standalone widgets, and the bridge wires it for semantic-mode params via `connect_changed`. Save (overwrite) and Refresh clear it.
  - **Refresh now works.** The Refresh action re-applies `mgr.active_preset` instead of `combo.itemText(currentIndex)`. The index-based version silently no-oped whenever the index was `-1` (after a session restore, or any state reset) — which read as "Refresh is broken" in the live app.
  - The combo shows a `"Presets…"` placeholder when nothing is active; `_reset_to_defaults` clears the active pointer + marker. `WidgetComboBox` already inherits header support from `ComboBox` (`add_header`/`header=`), but a header force-resets the index to -1 per pick — incompatible with the dirty marker — so a placeholder is used instead. Tests: `test_preset_manager.py` (+10: active persists across managers, restore-without-apply, marker set/clear on edit/save/refresh, Refresh-at-index-`-1` regression, placeholder, delete clears active; widget-state + semantic). pythontk `PresetStore.active` change ships alongside.

- **2026-06-18 Fix** — **`ExpandableList` child sublists no longer survive a hide/show with the list expanded.** Two distinct causes:
  - **Show side (synthetic Enter reopen):** `hideEvent`→`_force_hide_all` already collapsed sublists on hide, but Qt then delivers a synthetic `Enter` to the item under a stationary cursor when the list reappears, and that Enter ran the normal hover-to-expand path — silently restoring the previously-open sublist. Fix: a show-time latch (`_suppress_open_pos`, armed in the root list's `showEvent`) gates the Enter-driven open in `eventFilter` and clears the first time the cursor is seen at a *different* position, so genuine hover still expands but the show-time synthetic Enter is ignored. Direct/programmatic `_handle_widget_enter_event` calls bypass the gate (unchanged).
  - **Hide side (explicit dismiss did nothing):** `hide()` used to *early-return* while any sublist was visible (`_has_any_visible_sublist` guard). Because sublists are reparented to the window (not Qt children of the list), an explicit `list.hide()` then closed **nothing** — neither the list nor the sublist. `hide()` now force-collapses the hierarchy (`_force_hide_all`) and then hides; the now-unused `_has_any_visible_sublist` guard helper was removed.
  - Internal cleanup: the duplicated "resolve the root list" expression in `leaveEvent` now goes through a shared `_get_root_list()` helper.
  - Regression tests: `test_sublist_stays_collapsed_after_window_reshow`, `test_auto_open_resumes_after_cursor_moves`, `test_explicit_hide_collapses_open_sublists`.

- **2026-06-17 Fix** — **Marking-menu leaf clicks over Blender no longer go "dead" when the menu holds the mouse grab (`MarkingMenu` click-vs-chord classification + release-at-event-position).** Closes the leaf-click item left open by the grab-policy fix: a leaf — *Average* / *Set To Face* / … — needing up to four clicks, Blender only. **Reproduced + diagnosed live** (`tentacle/test/blender/normals_multiclick_check.py`, real Blender + the real buttons): with the grab held by the menu the named leaves fired **0/3** and the menu jumped to `cameras#startmenu`; with the fix, **3/3**. Mechanism: reaching `normals` through a *chord* (hold key + mouse button → main/edit → Normals) grabs the mouse via `_transfer_mouse_control`, so the menu — not the button — holds the grab. A leaf's left **press** is then delivered to `MarkingMenu`, where `_sync_menu_to_state` re-resolves `buttons=LeftButton` (key held) as the `F12|LeftButton` **chord** and navigates away before the leaf can fire. (In Maya the button consumes the press so the menu never sees it. Live, the activation poller's 0.02s `MouseTracking.track()` periodically migrates the grab to the button under the cursor, which lets a click through — so each click is a *race*, hence "up to four".) Two-part source fix: **(1)** `mousePressEvent` now treats a **lone-button press over an interactive descendant of the current start/submenu** as a click (early-return; dispatched on release via `_handle_widget_action`), never a chord — the discriminator is position + button count (`_is_menu_item_press` / `count_buttons == 1`), which already excludes a real chord (second concurrent button, or a press on empty overlay), so it correctly applies **even mid-chord** — that was the missing half (an earlier `_chord_gesture` gate wrongly skipped exactly the chord-reached case and is removed). **(2)** `mouseReleaseEvent` resolves the click target at the **release event's** position (`event.globalPos()`) instead of the live `QCursor.pos()`, which drifts off the leaf between the physical release and the pumped handler under Blender's timer-driven loop (equals the cursor under Maya's native loop → no behavior change there). Deterministic guard: `test/test_marking_menu_leaf_click.py` (leaf fires while the menu holds the grab; release dispatches at the event position despite a drifted cursor; genuine chord still navigates). Full marking-menu/overlay/events surface green; entire uitk suite 1882 passed.

- **2026-06-17 Fix** — **Marking-menu submenus no longer collapse to an ~18px sliver that renders detached from the gesture ("submenu shows in the wrong location") — `MainWindow.fit_height_to_content`.** A submenu's content (an `ExpandableList`, nested `Region`s) is **absolutely positioned** in a layout-less central, so the window exposes *no* layout height: both `minimumSizeHint().height()` and `sizeHint().height()` are `0`. The original `fit_height_to_content` hit `if target <= 0: return` and left such a window at its natural `_init_ui` size — correct. The recent `_content_min_height()` change (added to floor `adjust_height_by` against an under-reporting central) was also wired into `fit_height_to_content` *as the resize target*: it returns the tiny **laid-out** part only (`~18px` for a lone nav button), which is `> 0`, so the early return was skipped and the submenu was resized to 18px — its real content then overflowed/rendered outside the collapsed window. `_content_min_height` is now applied **only as a floor** once a positive target exists (preserving the overlap protection); when both hints are `0` the window keeps its natural size. Maya/standalone *and* Blender share this path, so both were hit. Reproduced live (fresh Maya, `cameras#lower#submenu`: 18px→600px, content renders in place) and locked headless: `test/test_main_window_resize.py::TestFitHeightToContent::test_fit_does_not_collapse_when_layout_min_is_zero`. Resize/footer/collapsable/marking-menu/menu/visibility suites 259 green.

- **2026-06-17 Fix** — **Clicking a marking-menu nav launcher now opens its submenu instead of raising (`MarkingMenu._resolve_button_menu` / `_cached_ui`).** The click path resolved a `MenuButton`'s **bare** `target` (`"cameras"`), but a bare-target nav launcher (`target="cameras"`, `filterTags="lower"`) isn't itself a registered UI — only its composed submenu `"cameras#lower#submenu"` is. So `_cached_ui("cameras")` → `get_ui("cameras")` raised `AttributeError("Slot class 'cameras' not found")`, which propagated out of `mouseReleaseEvent` (so the grab was never released either): clicking an upper/lower nav region threw instead of launching the submenu, and the failed click read as "needs several clicks". Now `_resolve_button_menu` resolves via the shared `Switchboard.menu_button_target_name` SSoT (also used by the `menu_button_target_resolves` auto-hide check, so the click and hide paths can't drift): the registered **base `target`** is preferred — preserving the original click→base-panel behavior (e.g. the polygons component buttons open the tag-filtered `polygons` panel) — falling back to the composed `submenu_name()` (`"cameras#lower#submenu"`) when the bare target isn't itself a registered UI (the cameras/editors nav launchers). `_cached_ui` now returns `None` for an unregistered name (honouring its `Optional` contract) rather than letting the resolver raise — every caller guards on a falsy result. Reproduced + locked headless against the real Blender menu (`cameras`/`editors#startmenu`): `test/test_marking_menu.py::TestNavButtonMenuResolution` (submenu vs standalone resolution, no-raise on unregistered, caching). Marking-menu suites 220 green.

- **2026-06-17 Fix** — **Marking-menu drag-time hover tracking is now host-aware so chord submenus reveal over Blender (`MouseTracking(buttons_provider=…)` + `MarkingMenu._host_mouse_buttons`).** A chord menu (`F12+LMB` → cameras, `F12+MMB` → editors) opens with the mouse grabbed (`_transfer_mouse_control`), and a Qt grab suppresses native `enterEvent`. The menu's `visible_on_mouse_over` `Region`s (the cameras/editors upper+lower nav clusters) depend on `MouseTracking.track()` to *synthesize* enter events for non-grabbable widgets during a grab — but `track()` is gated behind "a button is held", tested with `QApplication.mouseButtons()`, which is **blind to GHOST's mouse under Blender**. So during a Blender chord `track()` never ran, the Regions never revealed their nav buttons, and the upper/lower submenus "didn't launch" (and stale hover/grab state read as "click buttons more than once"). `MouseTracking` now takes an injectable `buttons_provider` (default `None` → queries `QApplication.mouseButtons()` **live at call time** — never cached, so patches/monkeypatches still apply); `MarkingMenu._host_mouse_buttons()` is the overridable hook wired into it (default = Qt, so Maya/standalone are byte-identical). The earlier speculative `mouseReleaseEvent` drop-grab (a wrong fix for the same symptom) is reverted — the grab is correctly *held* through the chord so `track()` can drive the reveal; it releases on the handled action / key-up as before. Live-verified on a fresh Blender (`tentacle/test/blender/cameras_track_check.py`: region reveals + hover opens `cameras#lower#submenu`); the Blender side that drives `track()` off its timer (GHOST never delivers `MouseMove` to the grabbed overlay) lives in `tentacle/tcl_blender.py`. Tests: `test_events.py::TestMouseTrackingDragHoverHandoff::{test_buttons_provider_overrides_qt_query, test_default_buttons_provider_reads_qt_live}`. Events/marking-menu/menu/resolver/visibility/option-box suites 262 green offscreen.

- **2026-06-16 Fix** — **Persistent widget settings no longer silently reset to defaults a few sessions later (`MainWindow.sync_widget_values`, `StateManager.save_value`, `Slots._process_deferred_widgets`).** Three independent ways a *default* could get written over a good value — each intermittent, so values "restored fine, then reset themselves" — are closed:
  1. **Cross-surface divergence (the main one).** A panel and its marking-menu surfaces (`<panel>`, `<panel>#submenu`, `<panel>#startmenu`) present the *same* controls but keep **separate per-surface QSettings namespaces** (`switchboard/<tagged_name>/…`). `sync_widget_values` only mirrored a change to a sibling surface that was **loaded *and* shown** at the time (`get_widget` reads `ui.widgets`, empty until a surface's first `showEvent`) — and it mirror-saved through the *current* UI's `StateManager`, i.e. into the wrong namespace. So a value set from the docked panel never reached an unopened marking-menu's store; reopening that surface a later session restored its stale/default copy. It now persists the value into **every** related surface's store directly (loaded or not) via the new `StateManager.save_value(key, value)` on a per-sibling branch (`_relative_state`, cached), and still mirrors onto any *live* sibling widget — but under that sibling's `suppress_save()` so the visual mirror can't ping-pong back into a save. (`sync_widget_values` now drives off relative *names*, so it no longer force-loads every sibling window on each change.)
  2. **Deferred-init default clobber.** The immediate `init_slot` path blocks a widget's signals around slot-init + restore; the **deferred batch** (`_process_deferred_widgets`) ran *unblocked*, so a widget whose `*_init` mutates its value (`addItems` flips a combo to index 0, `setChecked` fires `toggled`) persisted that post-init default **before** Phase-2 restore ran. Both phases now run inside `ui.state.suppress_save()`, matching the immediate path (restore-time slot firing is unchanged — still per `block_signals_on_restore`).
  3. **Combobox index transients.** `StateManager.save_value` skips a `-1` (no-selection) write for `currentIndexChanged` keys (a combo briefly reports `-1` while its model repopulates), and `apply` skips a stored index that outruns the current model (`value >= count()`) instead of letting it clamp/no-op and later persist the wrong index. Serialization is now a single chokepoint (`_coerce_for_store` + `save_value`, reused by `save`/`save_custom`).
  Regression tests in `test/test_persistence_hardening.py` (cross-surface store written for an unloaded sibling + restored on first open; deferred batch runs save-suppressed; `-1`/out-of-range combo guards). Persistence/state/switchboard/optionBox/marking-menu suites green (312 + 5 new).

The colour is re-applied on each `setStatusText`, so switching from an error back to a plain message cleanly drops the override (`_apply_status_level`). - **2026-06-16 Fix** — **Window auto-fit no longer shrinks into overlapping children when a central widget under-reports its minimum (`MainWindow._content_min_height`).** The first-show dead-space trim (`fit_height_to_content`) and the dynamic `adjust_height_by` both floored their resize on `self.minimumSizeHint().height()`. That hint is an unreliable floor: Qt's `qSmartMinSize` **replaces** a child's layout-computed minimum with any explicit `setMinimumSize` value — *even when the explicit value is smaller than the content needs*. So a central widget carrying a stale/too-small `minimumSize` baked into its `.ui` dragged the window's `minimumSizeHint` below the real content size, and the subsequent `resize()` packed the fixed-height children into too little space — they overlapped. Reported on the Scene Exporter panel under Blender's `windows11` style: `central_widget` advertised `minimumSize.height=240` while its content needed ~268, so the window snapped to 240 on show and the output group + footer overlapped the File box; the *expanded* output group was squeezed out of view, so it read as "always collapsing" even though its persisted state restored correctly. New `_content_min_height()` returns `max(self.minimumSizeHint().height(), centralWidget().minimumSizeHint().height())` — the central widget's own `minimumSizeHint()` reflects its layout's real minimum (it is not subject to the explicit-min override), so the floor always covers content while still trimming dead space above a footer/spacer. Both resize paths now floor on it. (The companion `mayatk` `scene_exporter.ui` also had its bogus `central_widget` min height zeroed.) Regression tests in `test/test_main_window_resize.py::TestFitRespectsTrueContentMin` (recover the true layout min past a too-small explicit central min; fit/adjust pin the window minimum at content height — deterministic, asserting `minimumHeight` not the live-WM resize outcome); window/widget suites 249 green offscreen.
- **2026-06-15 Fix** — **Menu-hosted dynamic widgets keep persisting after the menu reparents to a popup (`Menu._resolve_registration_window` / `_register_with_main_window`).** Widgets added via `Menu.add` (header-menu checkboxes/combos, option-box menu items) are registered with their owning `MainWindow` on a *deferred* event-loop tick (`_schedule_registration` → `QTimer.singleShot(0, _drain_pending_registrations)`). Registration found the window by walking `self.parent()` up to the first object exposing `register_widget`/`widgets`. But a menu **reparents itself to a top-level Tool/Popup window on show** (`setParent(active_window, Qt.Window)`), severing that chain — and in a live DCC session the show/reparent can win the race to run before the drain. The walk then hit no `MainWindow`, returned silently, and the widget never received `restore_state`, so its value was **neither saved nor restored** — the long-standing "menu-hosted options reset every session, but only in interactive Maya" symptom (headless/standalone/`mayapy.standalone` all passed because the reparent doesn't race the drain there). Fix: capture the owning `MainWindow` *weakly* while the chain is still intact — `_schedule_registration` now calls `_resolve_registration_window()`, which prefers a live parent-chain walk (cheap, correct in the common un-reparented case, refreshes the cache) and falls back to the cached window (liveness-probed; dead C++ wrapper drops the ref). `_register_with_main_window` routes through the resolver and now debug-logs the genuinely-unresolvable case instead of failing silently. `_restore_menu_defaults` gets the same cached-window fallback. Note: only widgets added through the menu path were affected — plain `.ui`-tree widgets register at panel-build time and were never at risk. Regression tests in `test/test_menu.py::TestMenuRegistrationSurvivesReparent` (live-chain resolve, cache fallback after reparent, and end-to-end add→reparent→drain still registers + sets `restore_state`); menu suite 150 green, state/persistence/option-box/mainwindow/switchboard suites green (1 pre-existing real-OS popup-width flake unrelated).
- **2026-06-15 Fix** — **`ExpandableList` now reopens collapsed — sublists no longer survive a hide/show in their previously-expanded state (`ExpandableList._force_hide_all`).** Sublists are reparented to the window (so they aren't clipped by intermediate native widgets), which puts them outside the root list's child tree. When an ancestor is hidden, such a sublist receives a *spontaneous* hide — `isVisible()` goes False — but keeps its **explicit-visible flag** (it was opened via `show()`, never explicitly hidden). `_force_hide_all` (run from `hideEvent`, and the hover-collapse paths) guarded its `QWidget.hide()` behind `if w.sublist.isVisible():`, so a sublist already reading as not-visible mid-hide was skipped, its flag survived, and Qt's `showChildren` re-showed it on the next show — the list reappeared expanded ("reshown in the previously open state sometimes"; which sublist had already gone non-visible by the time the root's `hideEvent` fired is what made it intermittent). The hide is now **unconditional** — the flag is cleared whether or not the sublist currently reads as visible — so the list always reopens collapsed; hiding an already-hidden widget is a no-op, so the extra calls are harmless across the shared callers (`hideEvent`, `_maybe_hide_sublist`, `_close_sibling_sublists`). Regression test in `test/test_expandable_list.py::TestExpandableList::test_force_hide_collapses_sublist_shown_under_hidden_ancestor` (shows a sublist under a not-visible window — `isVisible()` False, `isHidden()` False — then asserts `_force_hide_all` leaves it `isHidden()`; fails against the old guarded code); expandable/marking-menu/mouse-tracking suites 125 green.
- **2026-06-13 Fix** — **Option-box rows in the marking-menu overlay re-fit to content on show (`OptionBoxContainer.showEvent` / `_refit_to_content`).** A `tbXXX` option-box button in a `#submenu` overlay rendered far wider than its label — the option-box square pushed ~2–4× too far from the text — but **only on some hosts** (e.g. Blender's `windows11` Qt style; not Maya's `adskdarkflatui`). Root cause is a measure-before-polish + frozen-geometry interaction: the marking menu sizes overlay widgets via `center_widget(w, padding_x=25)` → `minimumSizeHint().width()+25` during `_init_ui`, **before** the theme QSS has polished the button, so on a style whose un-polished button minimum is large (Windows' native ~80px `CT_PushButton` floor) the row was resized to ~80px and that inflated geometry then froze in the absolute-positioned overlay (no parent layout re-runs to correct it once the QSS collapses the hint to ~32px). Styles whose un-polished minimum is already tight (Maya's) never inflated, so the bug was invisible there — and the docked panels masked it everywhere (their buttons stretch to the column regardless). Diagnosed by measuring the real overlays in fresh Maya 2025 vs Blender 5.1 (Blender "Clean" gap 28px / Maya 7px → both ~3px after). Fix: `OptionBoxContainer.showEvent` defers one tick (so the QSS has settled) then re-fits to its content size hint and re-centers — but **only when no parent layout owns it** (the absolute/overlay case); a layout-managed container, e.g. a docked panel where the button intentionally stretches, is left untouched. Verified end-to-end in fresh Maya + Blender GUIs (overlay gaps converge to ~3px; panels unchanged). Regression tests in `test/test_optionBox.py::TestOptionBoxOverlayRefit` (no-parent-layout container collapses an inflated size on show and preserves its center; a layout-managed one stays stretched); optionBox/switchboard 245, marking-menu/mainwindow 129 green.

- **2026-06-13 Fix** — **Persisted marking-menu bindings are now namespaced per host context (`MarkingMenu._binding_store_key` / `_bindings_store` / `on_bindings_changed`).** `SettingsManager` keys QSettings by a fixed `(org, app)`, so a Maya and a Blender session (same OS user) shared ONE `marking_menu_bindings` value — but the `F12+L+R` chord means `maya#startmenu` in one host and `blender#startmenu` in the other. Whichever wrote last hijacked the shared key, so once the forward-merge fix made Maya persist `maya#startmenu`, Blender's identical chord resolved to a UI it doesn't have (the reported "fix works in Maya, not Blender"). The store key is now suffixed with the host's `context_tags` (`marking_menu_bindings_maya` / `_blender`; standalone keeps the legacy un-suffixed key), so each DCC keeps an independent set — the pre-namespace key is intentionally orphaned (re-seeding each host from its own current defaults is the correct recovery). All five store accesses route through `_bindings_store`, and a new public `on_bindings_changed(callback)` lets the binding editor subscribe to the SAME namespaced store (tentacle's settings slots were poking the bare key directly — now they go through `bindings` / `on_bindings_changed`, verified live: the editor reads the identical namespaced dict the menu does). Verified end-to-end in a fresh Maya GUI (`store_key=marking_menu_bindings_maya`, L+R→`maya#startmenu`). Regression tests in `test/test_marking_menu_resolver.py::TestBindingStoreKey` (maya/blender keys differ, standalone unsuffixed, multi-tag deterministic).

- **2026-06-13 Fix** — **Marking-menu bindings now forward-merge new defaults into a user's persisted set (`MarkingMenu._reconcile_bindings`).** `__init__` seeded `configurable.marking_menu_bindings` only when storage was *empty*, so a user who had ever run an older build kept a frozen binding set and **never received newly-shipped default chords**. Concretely: the `F12+L+R → maya#startmenu` default was added after these users' bindings were persisted, so it was absent from the resolver's lookup and the chord fell through resolution step 2 (collapse-to-priority-button) to the RightButton-only sibling `main#startmenu` — "L+R no longer launches the maya windows." Now a new pure static `_reconcile_bindings(defaults, stored)` returns the dict to persist: `defaults` on first run, else `{**defaults, **stored}` (stored/user values win on key overlap, so rebinds survive while missing default keys are added), or `None` when the stored set already covers the defaults (no redundant write/`changed` signal).         plain         singleStep                    (default Qt step)
        Ctrl          singleStep × 10               (coarse)
        Ctrl+Shift    singleStep × 100              (very coarse)
        Alt           singleStep / 10               (fine)
        Ctrl+Alt      10 ** -decimals               (smallest)
