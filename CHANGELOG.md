# uitk — Changelog

## 2026

- **Widget-state restore hardening: checkable-button value path + no double-JSON-decode.** Two latent bugs in the state-restore value path (surfaced while auditing a "checkbox state not restored between sessions" report):
  - **`ValueManager.set_value` set a checkable button's *label* instead of its *checked state*.** Every `QAbstractButton` (incl. plain `QCheckBox`) inherits `setText`, and the `setText` branch preceded the `setChecked` branch — so a checkbox routed through the fallback `set_value` path had its text set to the value. The uitk `CheckBox` made it worse (it overrides `setText` with a rich-text setter), so `set_value(chk, True)` wrote `"True"` into the label and left the box unchecked. Added an `isinstance(widget, QAbstractButton) and widget.isCheckable()` branch (delegating to `_set_boolean_value`) *before* `setText`; non-checkable `QPushButton`s still take text. The signal path (`set_value_by_signal`'s `toggled`) was already correct — this only affects the fallback used by presets / pin-values / signal-less restore. Tests: `test_value_manager.py::TestSetValueCheckableButtons`.
  - **`StateManager.load` double-decoded values.** `MainWindow` passes a `SettingsManager` (not the raw `QSettings` the signature declares); `SettingsManager.value` already JSON-decodes, so `load`'s second `json.loads` was redundant and lossy (decoding an already-decoded `True` raised `TypeError` and silently fell back). `load` now only `json.loads` raw `str` values, passing already-typed values through — preserving the raw-`QSettings` path without the double-decode. Tests: `test_persistence_full_chain.py::TestStateManagerNoDoubleDecode`.
- **`MainWindow` snaps its height to content on first show (`fit_to_content_on_show`, default `True`) — removes the deadspace gap above a footer / trailing Expanding spacer.** Windows whose content is shorter than the shown height left an empty band between the last widget and the `Footer` (the trailing spacer absorbing the slack). `MainWindow.showEvent` now calls the existing `fit_height_to_content()` once on first show. The fit lives on `MainWindow`, not `Footer` — the footer is a fixed-height leaf with no authority over the top-level geometry, and `MainWindow` already owns the first-show lifecycle and the resize toolkit (`fit_height_to_content` / `adjust_height_by` / `_activate_descendant_layouts`). It is **skipped when a saved window size was restored**, so it never discards a size the user set intentionally: `restore_window_geometry()` now returns a bool (true only on a successful, non-degenerate restore) and `showEvent` passes on the fit when that's true. Opt out per-window with `fit_to_content_on_show=False`. Tests: `test_main_window_resize.py::TestFitToContentOnShow` (runs on first show by default, skipped when disabled, skipped when a saved size was restored, first-show-only, and an end-to-end "deadspace collapsed" check on the tentacle-style group+spacer+footer layout); `test_mainwindow.py::test_restore_disabled_ignores_saved_geometry_on_show` opts out so it keeps isolating the orthogonal saved-geometry path.
- **Preset selector: 2-column action grid, separator dead-space fix, and read-only built-in handling (`WidgetComboBox` + `PresetManager.wire_combo`).** Three related fixes to the persistent actions section under a preset combo, affecting every panel that uses preset templates:
  - **`WidgetComboBox.action_columns`** (default `1`) lays the action buttons out in a row-major grid. `PresetManager.wire_combo` sets it to `2` and reorders the actions to **Rename | Delete** on the first row and **Open | Save** on the second, replacing the four-line vertical stack.
  - **Separator dead-space fix.** The actions separator is a NoFrame/HLine that advertises an *invalid* sizeHint height, so `_apply_uniform_height` couldn't size its row and the view fell back to the delegate's font-based default — taller than the 7px line, rendering as visible dead space between the separator and the buttons (pronounced under Maya's larger UI font). `_rebuild_actions_section` now pins the separator row's `Qt.SizeHintRole` to the separator widget's own height.
  - **Read-only built-ins are marked and protected.** `wire_combo` italicises built-in (shipped) presets in the dropdown + adds a read-only tooltip (via the model item's `Qt.FontRole`/tooltip, so `itemText` stays the raw name for load/rename/delete, and the collapsed display keeps the widget font so the dropdown arrow is unaffected), and **disables Rename/Delete while a built-in is selected** — they can only no-op on a read-only preset, so greying them out is clearer than a silent click (the prior "deleting a built-in does nothing" confusion). Save stays enabled (writes a user preset that shadows the built-in — "duplicate to edit"), as does Open. Buttons follow their `QAction.setEnabled` state via the new `WidgetComboBox._bind_button_to_action` (bindings tracked + disconnected on rebuild so they don't accumulate). Tests: `test_widget_combobox.py::TestWidgetComboBoxActionsSection` (grid row-major, separator row pinned, button mirrors action-enabled, no connection accumulation) and `test_preset_manager.py::TestBuiltinTier::{test_wire_combo_italicises_builtins, test_wire_combo_disables_rename_delete_for_builtin}`.
- **`Header` menu button auto-hides when its menu has no real content, and auto-shows when content is added.** The header's `menu` (`☰`) button was always shown once configured, so tools that never populated their header menu (or only ever surfaced the transient "No options" placeholder) presented a button that opened an empty popup. The button now mirrors `Menu.contains_items` — which already excludes the empty-state placeholder — so an empty (or placeholder-only) menu keeps the button hidden, and populating the menu later reveals it. Implementation: `Header._sync_menu_button_visibility()` is the single source of truth (called from `showEvent`, from `config_buttons` after a layout rebuild that recreates the button visible, and wired to the menu's `on_item_added` signal for live auto-show); it reads `_menu` directly so it never force-creates a menu just to hide a button. This also surfaced and fixed a latent `Menu` bug: **`on_item_added` was never delivered to listeners** because `add()` blocks the menu's signals for the duration of the (recursive, bulk) insert, swallowing the synchronous `emit`. Added items are now queued and flushed once the *outermost* `add()` unwinds and the menu is settled (updates re-enabled, layout activated) — recursion tracked by a `_add_depth` counter so a single item, a bulk collection add (one ordered emit per item), and an externally pre-blocked `add()` (notifications dropped, never leaked into a later call) all behave correctly. Tests: `test_header.py::TestHeaderMenuButtonAutoVisibility` (hidden-when-empty, shows-on-add, placeholder-doesn't-count, never-accessed-stays-hidden) and `test_menu.py::TestMenuSignals` (delivery for single + bulk add, external-block suppression).
- **`WidgetComboBox` dropdown rows now sit tight with an exposed `item_spacing` (default 1px).** Rows showed an unexplained ~3px gap between embedded widgets. Cause: `_apply_uniform_height` captured each widget's `sizeHint` at *add-time*, but a widget's height can change afterward — a `CheckBox` measures 17px before the theme is applied and 14px after — so `_uniform_item_height` stayed at the stale 17 while the widget rendered at 14, and `AlignVCenter` rendered the 3px surplus as dead space between rows. Fixes: (1) `showPopup` now calls the new `_recompute_uniform_heights()` to re-derive the uniform height from the widgets' *current* sizeHints (rows fit tightly); (2) the deliberate inter-row gap is the new **`item_spacing`** property (default `1`, settable; row height = tight widget height + `item_spacing`), so the spacing is a single exposed knob instead of an accidental by-product of stale sizing. Set `combo.item_spacing = 0` for flush rows. Tests: `test_widget_combobox.py::TestWidgetComboBoxItemSpacing` (default, adds-to-row-height, stale-height correction); the existing uniform-height test pins `item_spacing = 0` to isolate that behavior.
- **`BridgeSlotsBase` log-panel `action://open` links now work standalone (no Maya).** `_on_log_link_clicked` delegated *every* `action://` anchor to mayatk's `UiUtils.dispatch_log_link`, so when a bridge panel runs as a standalone external app (the photogrammetry Metashape/RealityCapture/Brush case, and the common one) the import failed and a clicked output-dir link silently did nothing — even though the `open` action (reveal a file/folder) is DCC-agnostic. The base now handles `open` itself via the cross-platform `_open_in_file_manager` (and as a bonus this is cross-platform, where mayatk's handler used Windows-only `os.startfile`), and only the node-based actions (`select`/`reveal` in the Maya Outliner) defer to the lazily-imported Maya dispatcher. Additive + backward-compatible: the marmoset/substance/rizom `open?path=` links keep working and now also work outside Maya; node actions are unchanged. Covered by `test_bridge_slots.py::TestBridgeLogLinkOpen`.
- **`AttributeSpec` gains a `section` field; `BridgeSlotsBase` renders titled category dividers from it, plus a reusable `reveal_folder()` helper.** Parameter panels with many knobs were a flat list. A spec can now carry `section="…"`; `_build_param_widgets` inserts a titled `uitk.widgets.Separator` before the first spec of each new section (one per section; sections expected contiguous), and `_refresh_param_visibility` hides a section's divider when all its params are hidden for the current template/mode (so a fully-hidden category leaves no stray rule). Backward-compatible — specs without `section` render exactly as before (the DCC bridges set none). The field lives on the shared `AttributeSpec`, so `AttributeWindow` can adopt the same grouping when a consumer needs it. Also extracted **`BridgeSlotsBase.reveal_folder(path)`** (existence check + cross-platform open + error log) out of `open_templates_folder`, so subclass header actions ("Open Output Folder", …) reuse it instead of re-implementing. Covered by the extapps Metashape panel test `test_param_sections_have_dividers_that_track_mode`; the marmoset mock-bridge suite confirms the no-section path is unchanged.
- **Themeable layout metrics + uniform text inset across form widgets.** Two cosmetics that were hardcoded literals in `style.qss` are now exposed as theme tokens (resolved via the existing `{TOKEN}` substitution, editable live in the Style Editor):
  - **`COMBOBOX_ITEM_HEIGHT`** (default `19px`, was a hardcoded `min-height: 14px`) — the row height of combobox popup items. The token is the QSS `min-height` floor (also for plain `QComboBox` popups) *and* the value `_CurrentItemIndicatorDelegate.sizeHint` reads to **force** uitk `ComboBox` rows to exactly that height. The delegate enforcement is the key fix: `min-height` alone is only a floor, so a larger UI font or the combobox's `AdjustToMinimumContentsLengthWithIcon` icon-height reservation inflated rows past the token (the "rows look taller than 19px / uitk tallest" report). Rows carrying an explicit `Qt.SizeHintRole` are respected, so `WidgetComboBox` rows hosting a taller embedded widget still grow. The popup item's top/bottom padding is also zeroed so the plain-`QComboBox` floor isn't `token + 2px`. Tests in `test_combobox.py::ComboBoxPopupRowHeight` (default, custom value, clamp-down under a 20pt font, and explicit-size-hint respect).
  - **`StyleSheet.get_variable_px(name, theme, widget, default)`** — new SSoT accessor that parses a length token to an `int` (`"19px"` → `19`, `default` when unset/non-numeric). The delegate above and the Style Editor's px spin boxes now route through it instead of each re-implementing the parse (removed `style_editor._parse_px`). Covered by `test_style_sheet.py::test_get_variable_px`.
  - **`TEXT_INSET`** (default `2px`) — one horizontal text inset shared by `QLabel`, `QComboBox`, `QLineEdit`, and `QAbstractSpinBox` (symmetric left+right). Previously each set its own value (label 2px, combobox 1px, line edit 8px, spin box Qt-default), so text started at a different x in every widget; they now align in a layout. The combobox shifts ~1px right to match labels as requested; line edits tighten from 8px to the shared inset (tune the one token for roomier inputs).

  The Style Editor's `LENGTH_TOKENS` became a `{token: (min, max)}` map (SSoT for "which tokens are lengths *and* their spinbox range") so both new metrics render as px spin boxes rather than being misclassified as colors; `RADIUS`/`BORDER_W` keep their justified ≤8 cap. Tests: `test_style_editor.py::test_combobox_item_height_renders_as_spinbox` and `test_style_sheet.py::test_text_inset_is_uniform_across_form_widgets` (distinctive-override proof that each base rule reads the token).
- **`BridgeSlotsBase` gains a `_relevant_param_keys()` hook so panels can drive dynamic parameter visibility from something other than a template file.** `_refresh_param_visibility` hard-coded "the keys are whatever `__TOKEN__` placeholders the active template file references" — fine for the script-substitution bridges (marmoset / substance / rizom), but a single-runner panel selected by a *run mode* (e.g. the photogrammetry Metashape panel's Full / Align-only / Refine-only) had no template file and was forced to override the whole method to show every row statically. The "which keys are relevant?" decision is now a `_relevant_param_keys()` hook (default = the previous template-file behavior, returns `None` to skip), while `_refresh_param_visibility` keeps the shared row-toggle + height-fit bookkeeping. Run-mode panels override the hook to feed *their* current input (e.g. a run mode) to *their* `params_module.referenced_keys(...)` — so relevance stays declared in the params module (the SSoT) exactly as the script-substitution bridges declare it via `__TOKEN__` placeholders, with no relevance map leaking into the slots. Behavior-preserving for the existing bridges (verified by the marmoset mock-bridge suite + `test_bridge_slots.py`).
- **`ExternalAppHandler.launch()` self-heals a stale cached registration instead of failing with a confusing install-then-import error.** `discover()` caches each app's entry-point module path in `self._apps` once (at host startup). If that path later changes on disk — a package move (`extapps.metashape_workflow` → `extapps.photogrammetry.metashape_workflow`) or a version bump that rewrites an editable install's entry-point metadata — a long-running host (e.g. an open Maya session) kept importing the **old** path, hit `_is_importable() == False`, ran a surprise `pip install`, and then raised *"Install completed but `<old.module>` is still not importable"* — leaving the user stuck until a full host restart. `launch()` now, on import failure and only when the module came from the registry (no explicit `module=` kwarg), **re-discovers once and retries with the refreshed path** before the install fallback. Because `discover()` rebuilds a registration from entry-point metadata alone, the re-discovery **preserves the launch-augmentation fields a manual `register()` layered on** (`install_spec` / `python` / `show_kwargs`) — snapshotting them before and restoring any the rebuild blanked — so correcting the module path never silently drops the `install_spec` a host (e.g. tentacle's `install_spec="extapps"`) relies on for the install fallback. Importability is evaluated once and re-checked only after a successful self-heal, so the happy path and the explicit-`module=` install path are unchanged (existing `_is_importable`-call-count assertions still hold). Tests: `test/test_external_app_handler.py::TestExternalAppHandlerLaunch::{test_launch_self_heals_stale_registration, test_self_heal_falls_through_to_install_when_unchanged, test_self_heal_preserves_install_augmentation_fields}`. Verified end-to-end in a fresh Maya (via `mayatk.MayaConnection`): a stale `extapps.metashape_workflow` registration auto-corrected to `extapps.photogrammetry.metashape_workflow`, `install_spec` survived the heal, and the panel constructed (and found the local Metashape exe — no silent mock) without a host restart.
- **`PresetManager` + `BridgeSlotsBase` gain an opt-in *semantic-preset* mode — presets keyed by meaning, shareable with a headless CLI.** Bridge presets were raw widget-state keyed by `objectName` (GUI-only); a panel that wanted to persist the *same* run-template its CLI reads (e.g. photogrammetry's `specular_metal`) couldn't. Two additive, backward-compatible hooks:
  - **`PresetManager(value_provider, value_applier)`** (also on `setup()`): when both are set, `save` serializes `value_provider()`'s `{key: value}` dict (instead of widget snapshots) and `load` hands the payload (with `_meta` stripped) to `value_applier(data)` — so presets (de)serialize by semantic key. The two-tier `_store` (built-in + user via `pythontk.PresetStore`) and `wire_combo` are unchanged, so built-ins still layer under user saves and the combo's Save/Rename/Delete still work. Default (no callbacks) = exactly today's widget-state behavior.
  - **`BridgeSlotsBase.make_preset_store()`** (default `None` = widget-state, per-template subdirs under `PRESETS_ROOT`): returning a `pythontk.PresetStore` flips the panel to semantic mode — `_build_preset_controls` builds a `PresetManager` wired to `collect_param_values` / the new **`_apply_param_dict`** (maps `{param_key: value}` onto the panel's param widgets, ignoring keys it doesn't surface — overlay semantics matching a CLI `--preset`), and `_on_template_changed` no longer re-points the preset dir (the store is template-agnostic — one shared set). The DCC bridges (marmoset / substance / rizom) don't override the hook, so they're untouched. Tests: `test/test_preset_manager.py::TestSemanticPresetMode` (provider/applier round-trip, `_meta` stripped, user shadows built-in, combo lists both tiers) and the new `test/test_bridge_slots.py::TestBridgeSemanticPresets` (built-in run-template loads into param widgets, a Save is read back by the headless store, unknown keys ignored). This is the machinery the `extapps` Metashape panel now uses to share `specular_metal` + user presets with `run_combined --preset`.
- **`PresetManager` gains a built-in (shipped, read-only) preset tier and delegates storage to `pythontk.PresetStore`.** Presets were single-tier (user-writable only), so a panel couldn't ship default presets layered under a user's saves. `PresetManager` now takes an optional **`builtin_dir`** (constructor / `from_widgets` / `setup`) — a panel's `presets/` dir by convention, ignored when `None`/absent — whose presets appear in `list()` **under** the user tier (a user save of the same name shadows the built-in; saving a built-in's name is the natural "duplicate to edit" flow). New `source(name) → "user"|"builtin"|None` lets a UI lock/relabel built-ins; `delete`/`rename` act on the user tier only (built-ins are read-only). The file-facing internals (`list`/`save`/`load`/`delete`/`rename`/`exists`/`_preset_path` + name sanitization) now delegate to a `pythontk.PresetStore` built off the (still legacy-migrated) user `preset_dir` plus `builtin_dir`, so the GUI and any headless path share one implementation, directory layout, and naming — a user-saved preset is reachable both ways. **Backward-compatible**: callers that don't pass `builtin_dir` behave exactly as before (user-only); widget-state capture/apply and combo wiring are unchanged. Tests: `test/test_preset_manager.py::TestBuiltinTier` (built-in listed/sourced, loads into widgets, user save alongside + shadowing, read-only built-in).
- **`BridgeSlotsBase._selected_template_mode` accepts list-or-tuple `itemData`** — some PySide bindings round-trip the `(template, mode)` tuple stored via `addItem` back through `QVariant` as a *list*, so the `isinstance(data, tuple)` gate returned `None` and the send button reported "no template chosen." It now accepts either and normalizes to a tuple. Surfaced by the new standalone `extapps.marmoset_workflow` panel (the Maya-hosted bridges run under a binding that preserves tuples, so they never hit it). Strictly more permissive — can't break the tuple case.
- **`UiHandler.apply_styles` no longer treats the auto-installed help button as a configured layout** — its "did the slot already configure header buttons?" gate read `current = header.buttons.keys()`, and a slot's `header_init` calling `set_help_text(...)` populated `{"help"}` before `apply_styles` got a chance to apply defaults. The non-empty `current` made apply_styles skip its `config_buttons(...)` call entirely, so the UI was left with only the help button — the rest of the chrome (menu / collapse / hide / refresh) silently disappeared. Now the gate filters help out: `non_help_current = tuple(b for b in current if b != "help")`. If the only thing the slot installed is help, defaults still apply (and `config_buttons` preserves help across its rebuild because `_help_text` is set). Regression test `TestApplyStylesWithHelpButton` pins the three cases: help-only must NOT block defaults, non-help buttons MUST still block them, and empty headers get defaults as before.

- **`fmt()` rich-text tooltip helper gains `kbd()` / `hl()` and a `notes` parameter; colored title / section / note styling** — the helper used to render plain bold titles and white-text bullets, so multi-section help tooltips (Shot Manifest, Shot Sequencer, Channels, Mat Updater, etc.) ran together as a wall of text. Improvements:
  - **New `kbd(*keys)`** — render keyboard keys as `<kbd>`-style chips with monospace font, rounded border, and consistent spacing. Multi-key combos join with `+` so `kbd("Ctrl", "Z")` renders as two chips. Now used in Shot Sequencer (keyboard shortcuts section), Channels (wheel-scrub ladder), Wheel Rig (Shift-select), Color Manager (Ctrl-click).
  - **New `hl(text, color=_C_ACCENT)`** — wrap text in an inline color span for state callouts (e.g. <i>On</i> in green) without rebuilding the surrounding HTML.
  - **New `notes=[...]` parameter for `fmt()`** — italic muted "note:"-prefixed callouts rendered after main content. Use for caveats, plugin prerequisites, and "see also" hints.
  - **Color palette** tuned for Qt's dark tooltip background: titles in cool off-white, section headings in soft cyan, row keys / "note:" prefix in muted gray, note body in warm muted gold.
  - **Tighter spacing** — title `margin:0 0 3px 0`, sections `margin:6px 0 1px 0`, notes `margin:3px 0 0 0` — gives each segment visual breathing room without inflating the tooltip height.
  - 16 tests in `test/test_tooltip_mixin.py` covering chip rendering, color spans, each `fmt` section type, ordering invariants, embedded-HTML pass-through, and `kbd`-inside-`fmt` composition.

- **Header `set_help_text(...)` API replaces per-tool "Instructions" menu entries** — every Switchboard tool was duplicating the same 8-line pattern (`widget.menu.add("Separator", setTitle="About")` + `QPushButton` with `setText="Instructions"` and `setObjectName="btn_instructions"`) just to surface a tooltip nobody could find without opening the header menu first. `Header` now ships a `help` entry in `button_definitions` (icon: `uitk/icons/help.svg`) and a one-liner API:
  - `widget.set_help_text(text)` lazily inserts a `?` button at the leftmost slot (immediately right of the layout stretch) and stores `text` as its tooltip. Hover shows the tooltip (Qt default); clicking forces it open via `QToolTip.showText` anchored just below the button.
  - `widget.help_text()` reads the persisted text back; `widget.show_help()` is the click callback (also callable directly).
  - The text persists across `config_buttons(...)` rebuilds — config_buttons now re-adds the help button if `_help_text` was set but `'help'` wasn't in the list, so the help text never silently disappears when a later layout rebuild fires.
  - `'help'` is also accepted as an explicit `config_buttons` entry for callers who want fixed ordering.
  - 10 tests in `TestHeaderHelpButton` (auto-add on set, tooltip wiring, leftmost-slot insertion, idempotence, click-invokes-QToolTip, config_buttons persistence regression).
  - Migrated 39 ecosystem files away from the old menu pattern in parallel — see [`m3trik/docs/HEADER_HELP_INVENTORY.md`](../m3trik/docs/HEADER_HELP_INVENTORY.md) for the full list and the standalone-tool candidates that still lack help text.

- **Separator title no longer crops in narrow hosts** — `Separator` inherited `QFrame.sizeHint()`, which for the default `HLine` shape reports near-zero width. Once switched to `NoFrame` for the titled variant, the hint still didn't account for the child title label, so host containers (notably `QMenu`) auto-sized the separator off sibling widgets only — and any time the host was narrower than the title text, the label (a child positioned at `x=4` with no width clamp) was clipped on the right.
  - **Width fix**: `sizeHint()` and `minimumSizeHint()` now return `title_label.sizeHint().width() + 2 * margin` when the title is visible; `updateGeometry()` on title change makes parent layouts re-query. The 4-pixel positioning margin is extracted as `_TITLE_MARGIN_X` so the position math and the advertised width agree.
  - **Positioning fix** (follow-up): the width fix wasn't enough on its own — the label was still rendering cropped because `resizeEvent`'s gate was `isVisible()`, which is False whenever any ancestor is hidden. When a `Separator` lives in a menu that's hidden at construction (every `option_box.menu` is hidden until first show), no resize during the title-setter chain reached the positioning code, and the label stayed at its default 0×0 geometry. Even a correctly-sized host then rendered a 0-width label. Two changes: extracted `_position_title_label()` and called it directly from the title setter so the label gets its real size immediately; switched the guard from `isVisible()` to `not isHidden()` so subsequent resize events position correctly regardless of ancestor visibility. Regression test `test_title_label_has_non_zero_size_before_show` pins the contract.
- **MarkingMenu position stability — chord transitions no longer drift with cursor jitter** — every show of a startmenu used to call `overlay.start_gesture(QtGui.QCursor.pos())` and `setCurrentWidget` cursor-anchored. Repeated chord transitions (e.g. holding F12 and tapping LMB/MMB/RMB) rebound `start_pos` to whatever pixel the cursor was on at press time; natural hand jitter migrated the menu pixel-by-pixel in a consistent direction. User report: "the ui steadily shifts to the left or right a small amount each time."
  - Gesture origin is now established **once** per activation cycle. `_show_marking_menu`, `mousePressEvent`, and `keyPressEvent` only call `start_gesture` when `overlay.path.is_empty` (genuinely a fresh activation). Subsequent startmenu shows during the same cycle anchor at the preserved `start_pos` instead of the current cursor.
  - `setCurrentWidget` and `_prepare_ui` now accept an optional `anchor: QPoint` kwarg (defaults to current cursor for back-compat). `_return_to_startmenu` collapsed from prepare-then-override-with-move to a single `_prepare_ui(startmenu, anchor=start_pos)` call.
  - `_position_submenu_smooth` no longer side-effects the destination widget's size — uses the source's center for both `mapToGlobal` reads (mathematically identical to the resize-then-align approach).
  - 4 new tests in `test_marking_menu_position_stability.py` (initial cursor anchor, chord anchored at start_pos, zero drift across 20 jittered iterations, fresh gesture after path clear). Existing integration tests unchanged (one-line stub field added).

- **MarkingMenu breadcrumb clones now match the original launcher's size** — `Overlay._clone_widget` was silently dropping the size copy: `CLONE_ATTRS` listed `"size"` and the loop auto-generated the setter name as `setSize`, but `QWidget` has no `setSize` (the setter is `resize`). The `hasattr` check filtered it out and every clone fell back to Qt's default widget size (~100×30 for `QPushButton`). Breadcrumb clones placed along the gesture path looked oversized and didn't match the buttons they represented.
  - Fix: removed `"size"` from `CLONE_ATTRS` and added an explicit `new_widget.resize(prev_widget.size())` call *after* the attribute loop (so `minimumSize`/`maximumSize` don't clamp the final size).

- **MarkingMenu submenu positioning — resize no longer pulls neighbors off layout** — the real root cause of the "drift across most submenus" bug. `_position_submenu_smooth` resizes the destination's same-name widget (`w2`) to match the launcher's size; `w2.resize` keeps the top-left fixed, so its local *center* shifts by half the size delta. That shift propagated into `diff = p1 - p2` and `ui.move(ui.pos() + diff)` then slid every neighboring widget by the same amount — the visible symptom was clone-to-breadcrumb spacing tightening as you descended the hierarchy, even though the captured anchor positions were mathematically consistent. Fix: after `w2.resize`, also `w2.move` it back so its local center stays at the original layout position — the resize then affects only `w2`'s extent, not its position, and neighbors keep their layout offsets. New `test_smooth_positioning_resize_does_not_shift_neighboring_widgets` regression catches the shift directly.

- **MarkingMenu submenu positioning now anchors to path-saved center** — `_position_submenu_smooth` previously re-read the trigger widget's global center via `w.mapToGlobal(w.rect().center())` at smooth time, *after* `_prepare_ui` had already hidden the source UI.         Ctrl          singleStep × 10               (coarse)

        Alt           singleStep / 10               (fine)
