# uitk — Changelog

## 2026

- **Second-pass audit fixes (2026-06-11)** (`_audit/second_pass_audit_2026-06-11.md`): `ValueManager.set_value` no longer resets a numeric widget to `minimum()` on an unparseable value — the direct path now delegates to `_set_numeric_value`, which leaves the current value untouched (and accepts string numbers); `StateManager.save`/`save_custom` coerce `enum.Enum` values (PySide6's `Qt.CheckState` is not an int subclass, so the primitive-type gate silently dropped tri-state checkbox state — `stateChanged`-wired widgets now persist); `StateManager._defaults` switched to a `WeakKeyDictionary` (a plain dict pinned every registered widget forever) and `reset_all` skips + prunes wrappers whose C++ object is gone instead of crashing; duplicate `Menu.get_padding` removed (unused verbatim copy of the static `ExpandableList.get_padding`). Regression tests in `test/test_value_manager.py`.
- **Audit fix pass (2026-06-11)** (`_audit/uitk_audit_2026-06-11.md`): **`Switchboard.app` is now a lazy `ClassProperty`** — the old class-body `QApplication(sys.argv)` constructed an app as a side effect of merely referencing `uitk.Switchboard`, violating the package's no-side-effects-on-import rule (an app is still guaranteed once a Switchboard is instantiated); removed the duplicate `widgets.optionBox.options.clear` key in `DEFAULT_INCLUDE` (first entry was silently discarded); `Menu.add` no longer lets an item's objectName clobber Menu's own methods (an item named `clear`/`show` replaced the bound method silently — now guarded + warned); `AttributesMixin._set_custom_attribute` warns when a setter-looking kwarg (`set_*`/`setX…`) falls through to the plain-setattr fallback (typos used to be absorbed silently) and its error path logs instead of printing; `ui_history()` is now a pure read (it used to truncate + dedupe the stored history on every call — trimming moved to the `current_ui` setter); dead `Switchboard.__new__` removed; CLAUDE.md/`__init__` docs no longer point to the nonexistent `uitk/signals.py`.

- **Slot busy-cursor now auto-yields to *native* modal dialogs (`slots._ModalBusyCursorFilter`).** The earlier fix (below) only covered uitk's own modal helpers, which suspend the override around their `exec_()`. A slot that opens a **native** DCC dialog the switchboard doesn't wrap — Maya's `cmds.fileDialog2` / `promptDialog`, an OS file picker — still showed the busy hourglass over the dialog's own buttons and fields while waiting for the user (the reported symptom). Rather than wrap every native call site by hand, `SlotWrapper._invoke` now installs a `_ModalBusyCursorFilter` on the `QApplication` for the slot's duration (only while it holds the `WaitCursor`). Qt posts `QEvent.WindowBlocked` when a modal blocks the app and `QEvent.WindowUnblocked` when it's released — for native Qt-backed dialogs too — so the filter pops the whole override stack (preserving order) on the first block and restores it on the matching unblock; nesting is depth-counted so only the outermost pair toggles, and the hourglass resumes automatically for the slot's post-dialog work. The filter is observational (never consumes events) and defensively re-balances on cleanup if an unblock never arrives. uitk's own modal helpers keep their internal suspension (harmless belt-and-suspenders; their mocked-`exec_` test never fires a real block event). Tests: `test_switchboard.py::TestModalBusyCursorFilter` (block suspends / unblock restores, nested pairs, full-stack order, never-consumes, dangling-suspend cleanup).
- **Non-modal `text_view_dialog` now cancels the slot busy-cursor when it opens (`utils._drain_override_cursor`).** The earlier busy-cursor fix (below) suspended the override only around *modal* dialogs' `exec_()`, deliberately leaving `text_view_dialog` alone on the assumption it "self-corrects when the spawning slot returns." It doesn't reliably: the viewer is non-modal, so `show()` returns while the dispatcher still holds the `Qt.WaitCursor` override, and a slot that opens the viewer *then keeps working* (or streams content via `append_text`) leaves the hourglass sitting over the report. A suspend/restore context can't apply here — there's no bounded event loop to restore after, and re-pushing the WaitCursor on the window's close would strand a busy cursor with no matching pop (the reported "cursor stays active"). The correct behaviour for a non-modal viewer is to *cancel* the busy cursor outright: the slot's work product is on screen and the user is meant to interact with it. New `_drain_override_cursor()` pops the entire override stack after `show()`; this also makes the dispatcher's own `restoreOverrideCursor` `finally` a harmless no-op, keeping the stack balanced. No-op when no override is active (viewer opened outside a slot). Tests: `test_switchboard.py::TestSuspendOverrideCursor::{test_drain_clears_whole_stack, test_text_view_dialog_cancels_busy_cursor}`.
- **`Menu.sizeHint()` now reflects content width even while the menu is invisible — fixes the cropped option-box history popup (`Menu._activate_inner_layouts`).** A history/recent-values popup (e.g. tentacle's "Set Workspace" gear) is *sized then shown*: `RecentValuesPopup._show_popup` populates the menu, calls `adjustSize()`, positions it, then `show()`. But `add()` deliberately skips `self._layout.activate()` while the menu is off-screen (Qt re-activates before paint — see the bulk-add note), and only invalidates the *outer* layout. The inner `centralWidgetLayout` / frame `QBoxLayout`s kept the cached sizeHint from when they were empty — collapsed to just their margins `(4,2)` — while `gridLayout.sizeHint()` was correctly ~1100px. Since `Menu.sizeHint()` reads `self._frame.sizeHint()` directly (bypassing the grid's own recompute) and `invalidate()` only posts a *deferred* `LayoutRequest` that never runs while invisible, `adjustSize()` saw the collapsed width and clamped the popup to its 150px minimum — center-clipping every path into uselessness. `sizeHint()` now forces a synchronous `activate()` on the inner layout chain (`centralWidgetLayout` + frame layout; the outer `self._layout` is left to Qt, which drives its activation) **while the menu is invisible** — once shown, Qt drains the pending `LayoutRequest` and the hints stay current, so the hot visible path is byte-for-byte unchanged (verified: a shown menu reports the correct width with no activation). The refresh runs behind an `_activating_chain` re-entrancy guard, so every pre-show consumer (`adjustSize`, `resize(sizeHint())`, width-matching, screen-clamp) gets the true content width. The existing screen-width cap + `_clamp_to_screen` then keep a very wide popup on-screen as intended. Empty popups stay compact. Tests: `test_optionBox.py::TestRecentValuesPopupWidthSizing` (invisible-popup sizeHint ≥ grid content, `adjustSize` widens past 600px, empty stays < 400px) — all fail (sizeHint width 10) before the fix; full menu/optionBox suites (385) still green.
- **Modal switchboard-utils dialogs suspend the slot busy-cursor so they show natural cursors (`utils._suspend_override_cursor`).** `SlotInvoker._invoke` pushes an application-wide `Qt.WaitCursor` override for every slot's duration (the one busy affordance that survives a frozen Maya `cmds.*` call). A `QApplication` override cursor beats *every* widget cursor, so a modal dialog a slot spawns — `message_box(*buttons)`, `file_dialog` / `dir_dialog` / `save_file_dialog`, `input_dialog` — showed the hourglass over its own buttons, file list, and text field while *waiting for the user to interact*. A per-widget `setCursor` cannot win against an active override, so the fix suspends the whole override-cursor stack (pop-all, preserving order) for the dialog's `exec_()` and restores it exactly afterward via a `contextlib.contextmanager`; it's a no-op when no override is active (dialog opened outside a slot). Non-modal popups (`text_view_dialog`, passive `message_box`) are left alone — they don't block on interaction and self-correct when the spawning slot returns. Tests: `test_switchboard.py::TestSuspendOverrideCursor` (no-op without override, suspends+restores a single WaitCursor, restores a stacked override in order, restores on exception, and an end-to-end `input_dialog` check that `exec_` runs with the override cleared) — the `input_dialog` case fails before the fix.
- **Marking-menu navigation windows opt out of the on-screen clamp (`MarkingMenu.setCurrentWidget`) — fixes "the shift breaks the overlay starting position".** The clamp that pins a positioned menu fully within the anchor screen's available geometry now runs only when the widget opts in via `ensure_on_screen` (default `True`); it was previously unconditional. Stacked navigation menus (startmenu/submenu) already set `ensure_on_screen=False` in `_init_ui` (the same flag `MainWindow._ensure_on_screen` honors for standalone windows), so the clamp is now skipped for them and their center stays pinned to the **gesture origin**. That matters because a marking menu's directional flick is measured relative to that point — nudging the menu on-screen desynced the gesture and shifted the overlay's starting position; the user never expects or needs that shift mid-navigation. Standalone windows (shown via `_show_window` → `ui_handler.show`, never through `setCurrentWidget`) are unaffected and keep their `_ensure_on_screen` clamp. For navigation menus this restores the exact pre-clamp behavior (`widget.move(mapFromGlobal(anchor) - rect().center())`). Tests: `test_marking_menu_position_stability.py::TestSetCurrentWidgetOnScreenOptOut` (ensure_on_screen=False → unclamped center-on-anchor; True → clamped to the screen's available top-left; corner-anchored so it's screen-size-independent on CI 'offscreen'); the False case fails against the old unconditional clamp.
- **Marking-menu overlay now follows the active monitor — fixes "shows once, then subsequent shows silently fail" on multi-monitor (`MarkingMenu._ensure_fullscreen_on_active_screen`).** The overlay is a single frameless full-screen window and every menu is positioned inside it via `self.mapFromGlobal(...)`. Since `showFullScreen()` binds the window to whichever screen it currently occupies — set once at construction to the parent (Maya) window's screen — dragging the host DCC to a second monitor left the overlay pinned to the original screen. A menu anchored on the new monitor then mapped to local coordinates outside the overlay's bounds and never appeared (no error). `_show_marking_menu` now relocates the overlay to the screen containing the menu's **anchor** — the same point `setCurrentWidget` positions/clamps against (the gesture origin mid-chord, else the cursor), so the two always agree — (drop full-screen state → `windowHandle().setScreen` + `setGeometry(target.geometry())` → re-`showFullScreen`) **before** `setCurrentWidget` positions the menu, so the coordinate mapping resolves against the correct window origin. The relocation fires only on a *confirmed* screen mismatch (active screen ≠ overlay's `windowHandle().screen()`); single-monitor and indeterminate cases (no window handle / no screens) fall through to the original `if self.isHidden(): self.showFullScreen()`, so it's a strict no-op there. Regression: this is the `super().show()` → `self.showFullScreen()` change that introduced single-screen pinning. Tests: `test_marking_menu_multiscreen.py` (relocates on mismatch, no-op when already on the active screen, indeterminate falls back to show-if-hidden); existing `test_marking_menu*.py` (64 total) still green.
- **`Switchboard.add_reset_buttons(ui, widgets=None, *, types=(QAbstractSpinBox,), skip=(), **set_reset_kwargs)` — batch wrapper that gives a panel's value widgets per-field reset buttons in one call (`SwitchboardUtilsMixin`).** A thin loop over the option-box `ResetOption` (`widget.option_box.set_reset`): for every resolved widget it adds the small reset-to-default icon button (plain click resets to the registry default; Alt/Ctrl+click bypasses to default, greyed/restorable). Widget resolution **mirrors `connect_multi`** — `widgets=None` auto-discovers every child of `types` via `findChildren` (spin boxes by default; pass e.g. `(QComboBox,)` for a combo-driven panel), a shorthand string (`"s000-4"`) resolves through the naming convention, and an explicit list wires exactly those. `skip` accepts objectName(s) and/or widget instance(s) (e.g. a field sharing a tight row with a button); extra kwargs forward to `set_reset` (`reset=`, `icon=`, `tooltip=`, `on_toggled=`). Per-widget failures are caught + debug-logged (so one un-resettable widget can't abort the batch); returns the list of widgets that got a button. This is the shared SSoT for the pattern that several mayatk preview panels were about to hand-roll (it replaces curtain's private `_add_param_reset_buttons` loop). Tests: `test_switchboard.py::TestSwitchboardAddResetButtons` (findChildren discovery, string-pattern + explicit-list resolution, skip-by-name / skip-by-instance, `types=` combo override, kwargs forwarding, empty-container no-op) — uses uitk spin/combo widgets so it mirrors real panels.
- **Preset selector actions → a single row of compact icon buttons, plus a new Refresh (`WidgetComboBox` + `PresetManager.wire_combo`).** The two-row text-button grid (Rename | Delete / Open | Save) under a separator is now one toolbar-style row of icon-only buttons: **Rename · Refresh · Delete · Open · Save**. *Refresh* reloads (re-applies) the currently selected preset; the separator is removed. Backed by two general `WidgetComboBox` knobs: `action_icon_only` (compact, textless square buttons — text becomes the tooltip; default `False`) and `show_action_separator` (default `True`); every action button now also carries an `accessibleName` (its text) so icon-only buttons stay identifiable to screen readers (and tests). `wire_combo` sets `action_icon_only=True` / `show_action_separator=False` / `action_columns=len(actions)` and adds the Refresh action (icons edit/refresh/trash/folder/save at 16px). Tests: `test_widget_combobox.py::{test_icon_only_no_separator_single_row, test_separator_drawn_by_default}`, `test_preset_manager.py::{test_wire_combo_actions_are_icon_only_single_row_with_refresh, test_wire_combo_refresh_reloads_current_preset}`.
- **Centralized reset-to-default now follows an option-box reset toggle's bypass snapshot (`StateManager` + `ResetOption`).** When a field was *bypassed* (held at default via the reset toggle's Alt/Ctrl mode, which stashes a snapshot of the user's value) and the panel's global reset ran, the snapshot survived — so restoring the field resurrected the pre-reset value and the global reset effectively "didn't take." `StateManager.reset`/`reset_all` now call a duck-typed `widget.sync_stored_default(default)` hook right after applying the default; `ResetOption` registers it on the widget while bypassed (and clears it on restore), refreshing its snapshot to the new default. Decoupled by design — StateManager doesn't import the option. Tests: `test_persistence_full_chain.py::TestSyncStoredDefaultHook` (reset/reset_all invoke the hook; no-hook is harmless) and `test_optionBox.py::TestResetOption::test_centralized_reset_clears_bypassed_snapshot` (end-to-end: bypass → global reset → restore yields the default, not the stale value).
- **`SpinBox` / `DoubleSpinBox` gain `set_text_color()` (shared `SpinBoxTextColorMixin`).** Tints a spin box's displayed value text, enabling compact red/green/blue **X/Y/Z value coding** as an alternative to per-axis prefixes. The tint is applied as a `color` directive **on the spin box itself** — the uitk theme draws the value via the `QAbstractSpinBox` color rule, which overrides a color set on the embedded `lineEdit()` (verified by pixmap render under a competing theme rule), so the spin box is the only level that reliably wins. The directive is tagged with a `/*tc*/` marker and merged into (never allowed to clobber) any other inline style on the widget; `None`/`""` clears just our directive. The new mixin (`widgets/mixins/spin_box_text_color.py`) mirrors `wheel_step.py`'s shared-by-both pattern and is applied to both spin classes. Public API: `set_text_color(color)` + `text_color()`. Tests: `test_double_spin_box.py::TestDoubleSpinBoxTextColor` (default unset, applies to the spin box, merge-preserves other inline style, `None`/empty clears) and `test_spinbox.py::TestSpinBoxTextColor`.
- **New OptionBox plugin — `ResetOption` (per-widget reset-to-default button with a modifier-gated *bypass* toggle).** A small option-box icon button beside any value widget. A **plain click resets the field to its registry default** (a normal, *persisted* reset — the user chose it). Hold **Alt or Ctrl** while clicking to **bypass** instead: snapshot the current value, reset to default *transiently*, and grey the widget out; clicking the bypassed button restores the snapshot and re-enables. The icon goes the project "error" red while bypassed so held-at-default fields read at a glance. Bypass is **non-persistent** (each session starts un-bypassed — a panel never reopens with parameters mysteriously held at default); the snapshot is in-memory. The default is resolved **automatically** from the wrapped widget's window `StateManager` (`window.state.reset(widget)`) — the *bypass* reset runs inside the StateManager's `suppress_save()` (transient hold, doesn't bury the value at its default on close), while a *plain* reset persists. `widget.option_box.set_reset()` needs no per-field wiring — pass an explicit `reset` callable to override. Built as a first-class plugin: `options/reset.py::ResetOption(ButtonOption)` (plain `reset()` + `set_bypassed()` state, icon-swap visuals, `_current_modifiers()` seam for the Alt/Ctrl gate, emits `toggled(bool)` on bypass changes only), a fluent `OptionBoxManager.set_reset(...)`, and a `"reset"` slot in the option ordering (`_option_order` + `_sort_options` type map, between `pin` and `toggle`).
  - **Bypass no longer traps itself (`_optionBox.py` enabled-sync).** Bypass greys out the wrapped widget; the container's `_sync_option_buttons_enabled` (fired by the wrapped widget's `EnabledChange`) cascade-disabled *every* option button — including the reset button itself — so once bypassed it could never be clicked again to restore. Added a general per-button opt-out: `_sync_option_buttons_enabled` skips any option widget whose `keepEnabledWhenWrappedDisabled` dynamic property is set, and `ResetOption.setup_widget` sets it. Surgical — sibling option buttons (pin, clear, …) still cascade-disable with the wrapped widget; only the reset button is exempt (and Qt parent-propagation still effectively disables it when the whole container is disabled).
  - **Bypass snapshot/restore are now type-signal-correct (so a connected slot / live preview re-runs).** The restore set the value via the generic `ValueManager.set_value`, which for non-spin widgets could change the value without firing the signal the Switchboard wired for that type — a combo box restored by text needn't fire `currentIndexChanged`, and a check box's `get_value` even snapshotted its *label* (`text()` precedes `isChecked()`). Snapshot/restore now route through `ValueManager.get_value_by_signal`/`set_value_by_signal` keyed on the widget's `default_signals()` (falling back to the generic accessors for unregistered widgets). The reset-to-default path was already type-correct (it delegates to `StateManager.apply`, which uses the widget's default signal). Tests: `TestResetOption::{test_bypass_restore_emits_widget_type_signal_combobox, test_snapshot_uses_type_signal_checkbox, test_bypass_restore_emits_valuechanged_spinbox}`.
  - Tests: `test_optionBox.py` (`TestResetOption` — plain click resets without greying, `reset()`, Alt/Ctrl click enters bypass, click-while-bypassed restores, bypass button stays clickable while widget disabled + siblings still disable, snapshot/restore, `toggled`/silent-set signals, plain-reset-doesn't-emit, auto-reset via window `StateManager`, and the bypass-suppresses-save / plain-reset-does-not-suppress split; `TestOptionBoxManagerReset` — fluent returns-self/adds/replaces + the order key). All optionBox tests green (108).
  - Replaces the earlier `DisableOption`/`set_disable` (unreleased): the primary action is now a plain reset, with the bypass toggle behind a modifier, since the bare "undo" icon read as a simple reset. Downstream: `mayatk` curtain panel updated (`set_reset`).
- **Saved presets no longer vanish next session — legacy-wrap migration stops reburying placed packages (`PresetManager` migration block).** A user preset saved into a package dir (e.g. `mayatk/curtain`) disappeared on the next launch. The save/load core was fine; the deprecated folder-consolidation migration — which runs on the first preset op of every session via `preset_dir` → `_maybe_migrate_legacy` → `_maybe_clear_legacy_qt_roots` — was relocating the whole package tree into the inner `<root>/uitk/` wrapper, where the live load path never looks. Two bugs, two fixes: (1) **`_looks_like_ecosystem_wrapper` misfired on husks.** A prior wrap leaves behind data-less *husk* dirs at the root (`style_presets/`, `hotkey_presets/`, `switchboard_browser/` holding only a `.migrated` sentinel — `_merge_move` keeps the destination on a sentinel collision, so the source survives). Those non-known names made the detector return False *every* launch, re-firing the wrap forever. The detector now treats a husk (no `*.json` at any depth — new `_dir_has_preset_data` helper) as "already wrapped", same as an empty dir. (2) **`_wrap_pre_wrap_uitk_state` relocated known packages.** Even when it did run, it moved *all* root children — including correctly-placed `mayatk/` / `extapps/` siblings — down into the wrapper. It now skips any `known_pkgs` child (only uitk's own non-known pre-wrap dirs move down), so a freshly re-created package dir can never be buried. The two fixes are independent (either prevents the burial; together they also stop the needless churn) and each has its own regression test in `test_preset_manager.py::TestLegacyMigration`: `test_husk_dirs_do_not_rebury_known_packages` reproduces the field bug exactly (husks present → wrap must not fire), and `test_wrap_leaves_known_package_sibling_at_root` forces a genuine wrap (real pre-wrap state present) and pins that the known-package sibling is left at the root. Both fail before their respective fix and pass after; all existing pre-wrap / candidate-drain tests still green.
- **Disabled-while-checked buttons now read as disabled, not active (`style.qss` + `StyleSheet`).** A checked toggle that gets disabled (e.g. an option that doesn't apply to the current mode but is still checked) showed the full `BUTTON_CHECKED` accent — `:checked` and `:disabled` are equal-specificity, so the later `:checked` rule won and the control looked active. Added `QAbstractButton:checked:disabled` and `.default:checked:disabled` rules (two pseudo-classes → higher specificity) backed by a new **derived** token `DISABLED_CHECKED_BACKGROUND`: the resolved `DISABLED_BACKGROUND` blended ~22% toward `BUTTON_CHECKED` (new `StyleSheet._blend` 2-color lerp; `_disabled_checked_mix`), so the button reads as *disabled with a faint checked tint*. It's internal-only (injected by `_derive_internal_vars` alongside the `*_TINT` companions, tracks source/override changes, rejected by `set_variable` via `_is_derived_token`) — no new exposed theme var. Tests: `test_style_sheet.py` (`test_disabled_checked_background_derived_and_protected`; `test_qss_has_no_unresolved_tokens` now derives its known-token set the same way assembly does, covering any internal token).
- **Widget-state restore hardening: checkable-button value path + no double-JSON-decode.** Two latent bugs in the state-restore value path (surfaced while auditing a "checkbox state not restored between sessions" report):
  - **`ValueManager.set_value` set a checkable button's *label* instead of its *checked state*.** Every `QAbstractButton` (incl. plain `QCheckBox`) inherits `setText`, and the `setText` branch preceded the `setChecked` branch — so a checkbox routed through the fallback `set_value` path had its text set to the value. The uitk `CheckBox` made it worse (it overrides `setText` with a rich-text setter), so `set_value(chk, True)` wrote `"True"` into the label and left the box unchecked. Added an `isinstance(widget, QAbstractButton) and widget.isCheckable()` branch (delegating to `_set_boolean_value`) *before* `setText`; non-checkable `QPushButton`s still take text. The signal path (`set_value_by_signal`'s `toggled`) was already correct — this only affects the fallback used by presets / pin-values / signal-less restore. Tests: `test_value_manager.py::TestSetValueCheckableButtons`.
  - **`StateManager.load` double-decoded values.** `MainWindow` passes a `SettingsManager` (not the raw `QSettings` the signature declares); `SettingsManager.value` already JSON-decodes, so `load`'s second `json.loads` was redundant and lossy (decoding an already-decoded `True` raised `TypeError` and silently fell back). `load` now only `json.loads` raw `str` values, passing already-typed values through — preserving the raw-`QSettings` path without the double-decode. Tests: `test_persistence_full_chain.py::TestStateManagerNoDoubleDecode`.
- **`MainWindow` snaps its height to content on first show (`fit_to_content_on_show`, default `True`) — removes trailing vertical deadspace.** Covers both the gap above a `Footer` (when present — a trailing Expanding spacer absorbing the slack between the last widget and the footer) and the empty band at the bottom of a footerless window; most visible after a persisted session resize is restored larger than the content. `MainWindow.showEvent` now calls the existing `fit_height_to_content()` once on first show. The fit lives on `MainWindow`, not `Footer` — the footer is a fixed-height leaf with no authority over the top-level geometry, and `MainWindow` already owns the first-show lifecycle and the resize toolkit (`fit_height_to_content` / `adjust_height_by` / `_activate_descendant_layouts`). It runs **regardless of whether a saved geometry was restored**: a restored geometry's width and position are kept (the fit preserves width), and only the vertical dead space is trimmed. Opt out per-window with `fit_to_content_on_show=False` for windows whose extra height is meaningful (a growable list / scroll area). Tests: `test_main_window_resize.py::TestFitToContentOnShow` (runs on first show by default, skipped when disabled, runs even when a saved size was restored, first-show-only, and an end-to-end "deadspace collapsed" check on the tentacle-style group+spacer+footer layout); `test_mainwindow.py::test_restore_disabled_ignores_saved_geometry_on_show` opts out so it keeps isolating the orthogonal saved-geometry path.
- **Preset selector: 2-column action grid, separator dead-space fix, and read-only built-in handling (`WidgetComboBox` + `PresetManager.wire_combo`).** Three related fixes to the persistent actions section under a preset combo, affecting every panel that uses preset templates:
  - **`WidgetComboBox.action_columns`** (default `1`) lays the action buttons out in a row-major grid. `PresetManager.wire_combo` sets it to `2` and reorders the actions to **Rename | Delete** on the first row and **Open | Save** on the second, replacing the four-line vertical stack.
  - **Separator dead-space fix.** The actions separator is a NoFrame/HLine that advertises an *invalid* sizeHint height, so `_apply_uniform_height` couldn't size its row and the view fell back to the delegate's font-based default — taller than the 7px line, rendering as visible dead space between the separator and the buttons (pronounced under Maya's larger UI font). `_rebuild_actions_section` now pins the separator row's `Qt.SizeHintRole` to the separator widget's own height.
  - **Read-only built-ins are marked and protected.** `wire_combo` italicises built-in (shipped) presets in the dropdown + adds a read-only tooltip (via the model item's `Qt.FontRole`/tooltip, so `itemText` stays the raw name for load/rename/delete, and the collapsed display keeps the widget font so the dropdown arrow is unaffected), and **disables Rename/Delete while a built-in is selected** — they can only no-op on a read-only preset, so greying them out is clearer than a silent click (the prior "deleting a built-in does nothing" confusion). Save stays enabled (writes a user preset that shadows the built-in — "duplicate to edit"), as does Open. Buttons follow their `QAction.setEnabled` state via the new `WidgetComboBox._bind_button_to_action` (bindings tracked + disconnected on rebuild so they don't accumulate). Tests: `test_widget_combobox.py::TestWidgetComboBoxActionsSection` (grid row-major, separator row pinned, button mirrors action-enabled, no connection accumulation) and `test_preset_manager.py::TestBuiltinTier::{test_wire_combo_italicises_builtins, test_wire_combo_disables_rename_delete_for_builtin}`.
- **`Header` menu button auto-hides when its menu has no real content, and auto-shows when content is added.** The header's `menu` (`☰`) button was always shown once configured, so tools that never populated their header menu (or only ever surfaced the transient "No options" placeholder) presented a button that opened an empty popup. The button now mirrors `Menu.contains_items` — which already excludes the empty-state placeholder — so an empty (or placeholder-only) menu keeps the button hidden, and populating the menu later reveals it.
