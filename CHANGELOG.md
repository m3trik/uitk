# uitk — Changelog

## 2026

- **Current-item strip widened 1px → 2px + colour from theme `BUTTON_CHECKED`** — the 1px white strip read too subtle. Now 2px wide and sourced from the theme's `BUTTON_CHECKED` token (the orange "active selection" accent used by checked buttons elsewhere). Theme is resolved by walking up the combobox's parent chain to the nearest widget registered with `StyleSheet`, then `get_variable("BUTTON_CHECKED", theme=…)` is parsed via regex into a `QColor`. Hardcoded `rgb(165,135,110)` fallback if no theme can be inferred.

- **Row hover suppression scoped per-row in delegate, not blanket QSS** — the prior fix used a blanket `background-color: transparent` for `WidgetComboBox`'s view, which also killed the standard blue (`BUTTON_HOVER`) hover for **text-only** rows mixed into widget popups. Hover suppression now lives in `_CurrentItemIndicatorDelegate.paint`: it checks `view.indexWidget(index)` and strips `State_MouseOver`/`State_Selected` from the style option **only** when the row hosts an embedded widget. Text rows render with full hover styling; widget rows render with no row-level hover bg (avoiding the blue frame above/below shorter centred widgets).

- **`WidgetComboBox` row hover/selection bg suppressed** — after enforcing uniform-height rows + vertical-centering widgets, the parent QSS rule `QComboBox QAbstractItemView::item:hover { background-color: {BUTTON_HOVER}; }` showed as a **blue frame above and below** shorter embedded widgets (e.g. a 20px embedded combobox in a 24px uniform row). Embedded widgets already provide their own hover feedback and the current-item indicator strip still marks the active row, so the row-level shift was redundant. View's own stylesheet now sets `background-color: transparent` for `:hover`/`:selected`/`:selected:hover` at higher specificity than the parent rules. Text-only comboboxes are unaffected (different view, different stylesheet).

- **`WidgetComboBox` rows now uniform-height + zero item padding** — mixed-widget popups previously rendered ragged (slider ~22px / checkbox ~18px / spinbox ~20px) because each row's sizeHint came from the embedded widget's natural sizeHint. Added `_uniform_item_height` tracking that grows as taller widgets are added and re-syncs all existing rows so every selectable row reports the same height. The actions section (separator + multi-button container) is excluded via `track_height=False` so its legitimately taller container doesn't inflate normal rows. Embedded widgets are now added with `alignment=Qt.AlignVCenter` so shorter widgets stay at natural size centred in the uniform row instead of stretching. Item padding is overridden to `0` via the view's own stylesheet (parent `QComboBox QAbstractItemView::item { padding: 1px; }` was framing every embedded widget with a 1px gap unlike a standard combobox).

- **Current-item strip width 2px → 1px + colour from `palette.windowText()`** — after the popup item padding was tightened to `1px` uniform, the 2px strip overlapped the first 1px of every glyph in the current row (super().paint() drew text first, strip painted on top). Now `_STRIP_WIDTH = 1` so the strip fits inside the padding column and never touches text. Colour now sources from `palette.windowText()` (which Qt's QSS engine sets directly from the view's `color:` rule) with a fallback to `palette.text()` — both track the theme's text colour, so the strip is always visible against the surrounding background.

- **List/tree item-hover bg bumped from ~10% to ~20% alpha** — `QTreeWidget::item:hover:!selected` used `rgba(255, 255, 255, 0.1)` (float-alpha form Qt QSS parses inconsistently — often clamps to 0) and `QAbstractItemView::item:hover` used 30/255 ≈ 12% alpha. With dark theme defaulting to `BORDER_W: 0px`, the border-hover provided no fallback feedback, and the bg-hover at ~10% was imperceptible. Both rules now use integer `rgba(255, 255, 255, 50)` ≈ 20% — visible on dark, still subtle on light. Combobox popup items override with `BUTTON_HOVER` so they're unaffected.

- **Custom `_CurrentItemIndicatorDelegate` replaces the menu-style checkmark** — `SH_ComboBox_Popup == 0` killed the menu-item delegate, which also killed the checkmark next to the current item. New `QStyledItemDelegate` paints a 2px white-tinted accent strip on the left edge of the current row. The strip is drawn *on top of* the standard item paint, so item geometry is unchanged across all states — no text shift.

- **ComboBox popup forced to list-view rendering via `SH_ComboBox_Popup` override** — the previous indicator-zeroing in QSS didn't fix the hover text-shift because the shift wasn't from `QAbstractItemView::indicator`. With Fusion's default `SH_ComboBox_Popup == 1`, Qt installs a menu-item delegate (`QComboMenuDelegate`) that reserves a checkmark column on every popup item; that column is what offsets text right on hover/selected. `CustomStyle.styleHint` now returns `0` for `SH_ComboBox_Popup`, which keeps the plain `QStyledItemDelegate` and renders items edge-to-edge. The QSS indicator-zero rule stays as belt-and-suspenders.

- **ComboBox popup indicator column zeroed** — the residual ~12px text-shift on hover came from `QAbstractItemView::indicator { width: 12px; height: 12px; }` (the checkmark column for tree/table views). Fusion reserves that column on hover/selected items in combobox popups too, shifting text right. New rule `QComboBox QAbstractItemView::indicator { width: 0px; height: 0px; border: none; }` removes it inside combobox popups only — tree/table indicator behavior is unchanged.

- **ComboBox popup items get explicit padding + no-border** — after the Fusion swap, popup item text shifted noticeably right on hover (Fusion's `PE_PanelItemViewItem` paints selection chrome with internal padding the base state lacks) and the generic `QAbstractItemView::item:selected` rule added a 1px border that further offset text. New explicit rules: `QComboBox QAbstractItemView::item { padding: 2px 6px; border: none; min-height: 16px; }` and the same padding/border on `:selected`/`:hover`/`:selected:hover`, so item geometry is constant across all states. `min-height: 16px` also tightens vertical rhythm vs Fusion's roomier default.

- **ComboBox base style swapped from app-default to Fusion** — Win11's native style paints a blue accent border on focus that survives `outline: none` in QSS and `State_HasFocus` stripping at `CC_ComboBox` (Win11 paints the focus indicator through multiple primitive paths, not only the complex control). Win11 also delivers `:hover` events unreliably to popup items. Both behaviours go away when the `CustomStyle` proxy wraps `QStyleFactory.create("Fusion")` instead of `self.style()`. Visible chrome (drop-down arrow, frame, padding) is already QSS-painted, so swapping the base is cosmetically transparent. The popup view also gets `setStyle(self.custom_style)` + `setMouseTracking(True)` in `showPopup` so the same Fusion base + event delivery applies to the dropdown.

- **Bg-hover removed from `QFrame:hover` and `QAbstractScrollArea:hover`** — these are way too broad an attachment point: `QTextEdit`/`QPlainTextEdit` extend `QAbstractScrollArea` → `QFrame`, and the combobox popup is a `QListView` (same chain). The earlier `QTextEdit:hover { border: ... }` rule had no `background-color` declaration so the cascading bg from the broader rules leaked through unopposed, producing the textedit + combobox-view bg-hover the user reported. Both rules now keep only the border-hover; bg-hover survives only on widgets that are themselves a hover target (closed `QComboBox`, `QAbstractSlider`, `QProgressBar`).

- **Native focus-border suppressed on `ComboBox`** — Windows' platform style paints a blue focus border on `CC_ComboBox` whenever `State_HasFocus` is set, and `outline: none` in QSS does **not** cover this (QSS `outline` only affects QSS-drawn dotted rects). After clicking and moving the cursor away, the combobox retained focus and the native blue border lingered. Fix: `CustomStyle.drawComplexControl` (already wrapping every uitk `ComboBox`) now copies the option and strips `State_HasFocus` for `CC_ComboBox` before delegating, so the underlying platform style paints it as un-focused.

- **Combobox `:on` drops out of hover style** — when the popup opens, the cursor is still over the combobox, so `:hover` kept firing and the combobox's hover-bg (`WIDGET_BACKGROUND_HOVER`) sat directly above the popup's `WIDGET_BACKGROUND` — close-toned and adjacent, the two read as one continuous panel. `QComboBox:on` now explicitly resets background and border to the non-hover values; the popup itself is the open-state feedback.

- **Hover-bg dropped on text-entry widgets** — `QAbstractSpinBox`, `QLineEdit`, `QTextEdit`, `QPlainTextEdit` no longer change background-color on hover. The cursor lives in these widgets and a bg shift while moving the mouse competes with the text content. Border-hover (`BORDER_HOVER`) is unchanged. The five non-text widgets (`QComboBox`, `QFrame`, `QAbstractScrollArea`, `QAbstractSlider`, `QProgressBar`) keep the bg-hover as before.

- **Token rename `HOVER_BORDER_COLOR` → `BORDER_HOVER`** — aligns with the rest of the theme dict's `<FEATURE>_<STATE>` suffix convention (`WIDGET_BACKGROUND_HOVER`, `TEXT_HOVER`, `BUTTON_HOVER`, `TEXT_CHECKED`). The old state-prefix form was the lone outlier. 10 QSS sites updated.

- **Hover-bg + focus-rect fixes** — two artifacts surfaced after the prior bg-tint addition:
  - **Hover bg is now opaque**: the previous `rgba(255,255,255,30)` made inputs 88% transparent on hover, which revealed whatever rendered underneath (placeholder layers, sibling widgets, parent paint state) and surfaced as e.g. the widget's objectName overlaying its actual text. Replaced with a new `WIDGET_BACKGROUND_HOVER` token — opaque slightly-lighter per theme (`rgb(140,140,140)` light, `rgb(78,78,78)` dark) — so the bg change still reads as feedback but the widget stays fully opaque.
  - **Qt focus rect suppressed on inputs**: added `outline: none` for `QLineEdit`, `QComboBox`, `QTextEdit`, `QPlainTextEdit`, `QAbstractSpinBox`, `QFrame`, `QAbstractScrollArea`, `QAbstractSlider`, `QProgressBar`, `QAbstractButton`. Without this, Qt drew its native blue focus rectangle on widgets that had been clicked into and out of (e.g., the hierarchy manager's diff-options combobox), and the rect lingered after the cursor left.

- **StyleEditor + theme — six follow-ups** —
  - **Controls row gap removed**: dropped the `addSpacing(8)` between the theme combo and the show-tier label — the layout-default spacing now handles it consistently.
  - **`{RADIUS}` reach extended**: added `border-radius: {RADIUS}` to `Header`, `Header:hover`, `Footer`, `Footer:hover`, `QLabel`, and `QLabel:hover` (previously square-cornered with literal `0px`). The radius now affects these widgets uniformly with the rest of the UI.
  - **Radius cap tightened from 10 → 8**: empirical degradation point moved down after fixing the slider-handle and adding more `{RADIUS}` consumers; 8 is safe for every current `{RADIUS}` site.
  - **Dark theme defaults to `BORDER_W: 0px`**: the dark theme reads softer without internal widget borders by default. Light theme stays at `1px`. Users can still dial it up via the editor.
  - **`ColorSwatch` gets a fixed black border**: previously `border: 8px solid transparent` (faded into the panel for light fills); now `border: 1px solid black`. `border-radius` and the checked-state overlay (`border-color: <fill>`) are unchanged from the original.

- **StyleEditor — layout + radius-cap polish** —
  - **Controls row spans the window**: the Theme/Show comboboxes now stretch (factor 1 each) so they fill the row width instead of clumping on the left.
  - **Length spinboxes capped at 10**: above 10 Qt's border-radius rendering degrades on small widgets. Cap applies to both `RADIUS` and `BORDER_W` for simplicity.
  - **Spinbox container loses horizontal padding**: cell QSS padding (`2px 5px`) already provides visual breathing room; the extra container `4px` was contributing to the cramped feel.
  - **Slider handle decoupled from `{RADIUS}`**: `QAbstractSlider::handle` is `width: 10px` by design, so scaling its corner with the global radius turned it into a pill at 6px and a degraded oval at 10px+. Now uses a literal `1px` like the scrollbar handle. Locked in by a new `test_slider_handle_radius_is_literal_not_token` test.
  - **Spinbox widened to 76px**: Qt's QSpinBox reserves internal layout space for the up/down arrow column even when QSS hides those buttons; 60px cropped the visible " px" text.

- **StyleEditor — live self-styling, length-token inputs, tier filter** —
  - **Live self-styling**: the editor now registers itself with the theme system on `__init__`, so color/size edits show up in its own chrome in real time. The theme combobox both filters the table and re-themes the editor — switching to "dark" makes the editor go dark and the rows reflect dark theme values. Registered with `recursive=False` because Qt stylesheets cascade to children naturally; recursing would wipe `ColorSwatch`'s inline self-styling.
  - **Length-token inputs**: tokens listed in `LENGTH_TOKENS` (currently `RADIUS`, `BORDER_W`) render with a `QSpinBox` + `" px"` suffix instead of a `ColorSwatch`. Length and color rows are separated by a bold `— Sizes —` divider row that spans all columns.
  - **Tier filter**: new "Show:" combo with `Basic` (12 most-edited tokens — main surfaces, text, accents, border color, radius) / `All` (every token, including the `ACTION_*` status palette and Python-consumed `ICON_COLOR` / `LINK_*`). Defaults to Basic.

- **StyleSheet — token cleanup + parsed-template engine** — finalized the dynamic-stylesheet token set and replaced the per-token `str.replace` loop with a parsed-template engine.
  - **Engine**: `style.qss` is now split once on first load into `[literal, TOKEN, literal, TOKEN, ...]` (cached as `_template_cache`), and `_apply_template` assembles each QSS string with one `''.join` of dict lookups instead of N full-string scans. `reload()` groups registered widgets by `(theme, resource, package, kwargs)` and assembles each group's QSS once — large registries no longer pay the template-apply cost per widget. Dropped the stale `setStyleSheet("")` clear in `_set_style` (Qt diffs internally). New `StyleSheet.clear_caches()` helper for dev hot-reload.
  - **Token additions**: `RADIUS` (default `1px`, replaces ~20 literal `border-radius: 1px` defaults), `BORDER_W` (default `1px`, replaces every `1px solid …`), `HOVER_BORDER_COLOR` (replaces the 10 literal `rgba(255,255,255,110)` :hover borders). Semantic-exception literals (footer `3px`, scrollbar `2-4px`, optionBox `0px`, 2px focus emphasis) intentionally stay inline.
  - **Renames**: `TEXT_BACKGROUND` → `SELECTION_BG`, `HIGHLIGHT_COLOR` → `SELECTION_FG` (text-selection bg/fg in inputs; the originals were misnamed), `WINDOW_FOREGROUND` → `TREE_ALTERNATE_BG` (tree alternate-row tint; the original was misleading since FG usually means text).
  - **Drops**: `TABLE_ITEM_HOVER`, `TABLE_ITEM_SELECTED`, `BUTTON_PRESSED` — zero meaningful QSS uses; `BUTTON_PRESSED`'s one site was a scrollbar arrow that takes the literal `rgb(120,120,120)` inline.
  - **Typo fix**: `{PANEL_BACKGROUND_BACKGROUND}` (doubled suffix on Footer) → `{PANEL_BACKGROUND}`.
  - **Breaking**: persisted user overrides bumped from `overrides_v2` → `overrides_v3` namespace; no migration — old `v2` data remains on disk but unread. Any external Python calls to `get_variable("TEXT_BACKGROUND" / "HIGHLIGHT_COLOR" / "WINDOW_FOREGROUND" / TABLE_ITEM_* / BUTTON_PRESSED)` will return `""` and need to be updated. `LINK_COLOR` / `LINK_VISITED_COLOR` kept (consumed by `textEditLogHandler.py`).
  - Net token count: 31 → 31 (drop 3, rename 3, add 3) — cleaner semantics, same surface size.
  - New tests: template engine cache behavior, alternation parsing, unknown-token fallback, QSS-tokens-resolve-against-themes sanity check, and a grouping-optimization assertion that 5 widgets sharing a config trigger 1 template-apply call (not 5).

- **OptionBox dead-code purge** — removed unused surface from `OptionBoxContainer` and `OptionBoxMixin`. Deleted: `OptionBoxContainer.setPassThrough` / `_pass_through` / `_update_mask` / `resizeEvent` (no callers anywhere in the monorepo); `OptionBoxMixin.has_pin_values` and `get_pin_values_service` (both methods had zero call sites and both depended on `_pin_values_controller`, which has zero writers — they unconditionally returned `False`/`None`). Also dropped the no-op parent-walk branch in `OptionBoxManager._find_existing_option_box` (returned `None` on hit). No behavior change.

- **Renamed ExternalToolHandler → ExternalAppHandler** — The handler that installs/launches uitk-compatible Python packages is now uniformly named after the things it launches (apps, matching the `extapps` package). Affected surface: module file `external_tool_handler.py` → `external_app_handler.py`, class `ExternalToolHandler` → `ExternalAppHandler`, handler key `external_tool` → `external_app`, entry-point discovery groups `uitk.external_tools[.in_process]` → `uitk.external_apps[.in_process]`, `CONFIG_BRANCH` (QSettings branch for persisted user_tags) `"external_tool"` → `"external_app"`, internal cache `_tools` → `_apps`. The `kind` strings reported in `HandlerEntry` (`external_in_process` / `external_subprocess`) and the browser's `_KIND_CHIP_LABELS` were left as-is — those describe the launch-mode taxonomy, not the handler. **Breaking change**: callers must update handler-key references (`sb.handlers.external_tool` → `sb.handlers.external_app`) and imports. Persisted user_tags under the old QSettings branch are dropped on first run.

- **ExternalAppHandler — `launch(show=False)` + install-failure wrapping** — `launch()` previously always called `widget.show() / raise_() / activateWindow()` after import, leaving no hook for hosts that want to inject context (e.g. `ui.slots.texture_provider`) and route the widget through a custom show path (such as `marking_menu.show()` for cursor-relative positioning). Added a `show: bool = True` kwarg: when False, the widget is install-checked, imported, cached, reparented under the host, and visibility-wired — but not shown. Refactored `_show_in_process` into `_prepare_in_process` (parent + visibility filter) + `_show_in_process` (prepare then show/raise/activate). Also wrapped `PackageManager(...).install(...)` so any underlying pip / OS failure surfaces as a single `RuntimeError` (was previously letting raw OSError/subprocess errors propagate), giving hosts a single exception type to catch. New tests in `test_external_app_handler.py`: `TestInProcessShowOptOut` (3 cases) and `TestInstallFailureSurfaces`.

- **SpinBox / DoubleSpinBox Ctrl+Alt wheel fix** — Ctrl+Alt+scroll on either widget now steps the value by the lowest decimal place (10⁻ᵈᵉᶜⁱᵐᵃˡˢ), as intended; previously it did nothing on PySide6.
  - **Root cause**: `wheelEvent` polled `QGuiApplication.keyboardModifiers()` and dispatched with strict `==`. In PySide6, `Qt.ControlModifier | Qt.AltModifier` evaluates to a `KeyboardModifier` value while the live modifier state is a `KeyboardModifiers` flag set — `==` between them is false even when bits match, so the Ctrl+Alt branch never fired. `SpinBox` also had unreachable dead code inside the Ctrl-only branch claiming to handle Ctrl+Alt. `DoubleSpinBox` additionally had Ctrl-alone and Ctrl+Alt bindings swapped relative to `SpinBox`.
  - **Fix**: read modifiers from `event.modifiers()` (event-time, also unit-testable) and detect with bitwise `&`. `decreaseValueWithSmallStep` now always uses `10 ** -decimals` instead of `max(step/10, 10**-decimals)`, so Ctrl+Alt truly targets the lowest decimal place regardless of `singleStep`. `DoubleSpinBox` bindings aligned to `SpinBox`.
  - 5 new wheelEvent dispatcher tests in `test_spinbox.py` plus a new `test_double_spin_box.py`.

- **CollapsableGroup ↔ MainWindow resize fix** — collapsing a `CollapsableGroup` inside a uitk `MainWindow` no longer leaves the window oversized.
  - **Root cause**: QMainWindow's auto-cached `minimumSize` follows `minimumSizeHint` UP when content grows but does NOT follow it back DOWN when content shrinks. After a collapse, `minimumSizeHint` correctly dropped to e.g. 156 but the explicit `minimumSize` cache stayed at the pre-collapse 279 — so the subsequent `resize()` got clamped above the requested shrink. Observed in tentacle: window shrunk by ~10px when the group lost ~120px of content.
  - **Fix**: `MainWindow._sync_min_height_to_hint()` aligns `minimumHeight` with the freshly-computed hint before each `resize()` in `adjust_height_by` and `fit_height_to_content`. `CollapsableGroup._fallback_window_resize` does the same for non-MainWindow hosts. Tracks in both directions: once we take over the explicit min, Qt stops auto-raising it on grow, so subsequent expands need us to raise it back too.
  - Added 5 end-to-end regression tests in `test_main_window_resize.py` exercising a tentacle-style layout (nested QVBoxLayouts, multiple `CollapsableGroup`s, trailing `Expanding` spacer, footer).

- **ToggleOption** — new OptionBox plugin (`uitk/widgets/optionBox/options/toggle.py`) for persisted binary on/off buttons. Icon dims to the project's "error" red (`pythontk.Palette.status()["error"]`) when off, so the user can see which control caused a dependent filter / widget to stop.
  - Emits `toggled(bool)` on user clicks and on `set_on(v)` (suppress with `emit=False`).
  - Optional `gated_widgets=[…]` disables listed widgets in sync with the toggle. Does not auto-restore on destruction — caller owns widget lifecycle.
  - `OptionBoxManager.set_toggle()` / `add_toggle()` for fluent setup, including `on_toggled=` to wire the signal in one line.
  - Persistence: pass `settings_key=str` for explicit namespacing, omit to auto-derive from the wrapped widget's `objectName`, or `False` to opt out (consumer owns storage).
  - Default `_option_order` now includes a `"toggle"` slot between `"pin"` and `"action"`.
- **Persistence mixin** — extracted shared `settings_key` resolution + lazy `SettingsManager` construction into `optionBox/options/_persistence.py` (`PersistedOption`). Currently consumed by `ToggleOption`; `ActionOption`, `PinValuesOption`, and `RecentValuesOption` keep their existing ad-hoc copies and will be migrated in a follow-up.
- **switchboard_browser filter button migration** — `_build_filter_lineedit` now uses `ToggleOption` instead of a hand-rolled two-state `ActionOption`. The fragile `find_option(ActionOption)` lookup at the end of the factory ("the first action that isn't the scope action") is gone — the toggle reference is held directly from construction. Preset restore now syncs the toggle's visual state via `set_on(..., emit=False)` so the icon reflects the restored `filter_enabled` flag.

- **Menu popup fixes** — fixed option menus not opening and flashing during init.
  - **Root cause**: `_register_with_main_window()` called synchronously inside `Menu.add()` triggered `mainWindow.register_widget()` → `widget.init_slot()` → `tb###_init()` recursively during menu population. Fixed by deferring via `QTimer.singleShot(0)`.
  - `showEvent`: skip `_apply_position()` when `show_as_popup()` already handled positioning (check `_current_anchor_widget`).
  - `_setup_as_popup`: consolidated double `setWindowFlags()` + `setParent()` into single `setParent(parent, flags)` — avoids double native window recreation.
  - `Header.show_menu`: changed `menu.setVisible(True)` → `menu.show_as_popup(anchor_widget=self, position="bottom")`.
  - `MenuOption` / `OptionMenuOption.set_wrapped_widget`: preserve popup flags via `setParent(widget, self._menu.windowFlags())`.
  - Added 13 regression tests in `test_menu.py` covering init visibility, deferred registration, header menu popup, option box menu flags.

- **Marking menu reshow fix** — fixed marking menu reopening on top of standalone windows while activation key is still held.
  - `_show_window`: always hides the MarkingMenu and clears `_activation_key_held`; reparents child widgets so they aren't hidden alongside the overlay.
  - `_transition_to_state`: guards against transitioning when MarkingMenu is already hidden, preventing `Key_F12` → startmenu reshow.
  - `_standalone_suppress` flag: set in `_show_window`, cleared in `_on_activation_release`. Blocks `_on_activation_press` from processing spurious key re-fires caused by focus changes in Maya.
  - Added 9 regression tests in `test_marking_menu.py`.
